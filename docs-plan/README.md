# how-to-write-docs

本文件夹用于**指导 AI 如何规范地撰写开发文档，并按照固定结构与用户进行设计讨论与开发推进**。

该规范适用于 iris-code 及其子项目，目标不是追求形式完整，而是确保：
- 设计思考顺序正确
- 模块边界清晰
- 后续开发与讨论具备一致的工程语境

---

## 一、核心原则（AI 必须遵守）

### 1. 文档 ≠ 结果，文档是思考过程的外显
AI 在撰写任何开发文档前，必须与用户讨论设计假设，而不是直接生成“最终答案”。

### 2. 严格按照文档顺序推进，不得跳步
在一个模块的设计中，文档必须按以下顺序进行讨论和产出：

1. goals-duty.md  
2. architecture.md  
3. data-model.md（可选、精简）  
4. dfd-interface.md  
5. test.md  

除非用户明确说明，否则不得跳过或提前编写后续文档。

### 3. 每一阶段都应与用户确认
AI 在每一类文档完成后，应主动向用户确认：
- 是否符合预期
- 是否需要调整目标、边界或设计方向

---

## 二、AI 的工作方式要求

### 1. 先讨论，再落文档
在生成任何 `*.md` 文件前，AI 应：
- 提出结构化问题
- 引导用户明确设计意图
- 基于讨论结果再生成文档草案

### 2. 文档是“模块级”的，而不是“项目级”的
本规范中的文档，默认针对**单一模块**，而不是整个系统。

模块是 iris-code 中最重要的设计单元。

### 3. 避免过度设计
如果信息不足，AI 应：
- 明确标注假设
- 使用“建议方案”而非“强制结论”
- 保持文档可演进性

---

## 三、各文档的角色说明（概览）

| 文档 | 作用 |
|----|----|
| goals-duty.md | 明确模块存在的目的与职责边界 |
| architecture.md | 说明模块内部结构、设计模式与文件组织 |
| data-model.md | 描述核心抽象（精简、非强制） |
| dfd-interface.md | 描述数据如何流动，以及模块如何对外交互 |
| test.md | 说明如何验证模块在协作中的正确性 |

**具体如何撰写，请参考同目录下的 `*-guide.md` 文件。**

---

## 四、与用户讨论时的基本姿态（非常重要）

AI 在与用户讨论设计时，应遵循以下准则：

- 不抢先做设计决策,但提供设计建议
- 不默认“用户需求是完整的”
- 不为了“写得快”而牺牲结构清晰度
- 始终围绕「模块」这一核心节点展开讨论
