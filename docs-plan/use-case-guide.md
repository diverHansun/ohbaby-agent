# use-case-guide.md

本指南用于指导 AI 如何为**单一模块**撰写 `use-case.md`，并在撰写前与用户确认模块内部的**业务动作、执行编排与责任落点**。

`use-case.md` 的核心任务不是描述实现细节，  
而是回答：

- 当前模块要完成哪些关键业务动作？
- 每个动作是如何从输入推进到输出的？
- 哪些步骤属于当前模块负责，哪些属于外部模块或基础设施？

它用于补足 `goals-duty.md` 与 `dfd-interface.md` 之间的空档，  
使“职责”能够落到“可执行的业务流程”上。

---

## 一、use-case.md 的核心作用

`use-case.md` 定义**模块内部如何围绕职责完成关键业务动作**。

> **use-case.md 的目标是明确”职责如何被执行”，而不是提前写实现。**

---

## 二、何时需要 use-case.md

AI 在以下情况下**建议撰写** `use-case.md`：

- 模块不只是简单工具函数，而存在明确的业务动作
- 模块内部包含多个步骤的协调、编排或状态推进
- 仅靠 `goals-duty.md` 无法说明“职责如何真正落地”
- 后续实现中容易出现 controller / service / domain / adapter 职责混淆

AI 在以下情况下**可以省略或极简**：

- 模块非常简单，仅完成单步转换
- 不存在明显的业务动作或流程编排
- 该模块本质上只是纯函数工具层

如果选择省略，AI 应向用户说明理由。

---

## 三、撰写前的前置条件

在开始撰写 `use-case.md` 之前，AI 必须确认：

1. 模块的职责边界已在 `goals-duty.md` 中明确
2. 模块的内部结构已在 `architecture.md` 中有基本说明
3. 若存在 `data-model.md`，其核心概念可被引用
4. 若模块涉及外部交互，`dfd-interface.md` 已能说明输入来源与输出去向

如果这些前置条件不成立，AI 不应直接撰写，而应先引导用户补齐前置文档。

---

## 四、AI 在撰写前应与用户澄清的问题

在生成 `use-case.md` 之前，AI 至少应与用户确认：

- 当前模块最关键的 1~3 个业务动作是什么？
- 每个动作的输入来自哪里？
- 当前模块真正负责哪些步骤，哪些只是在调用外部能力？
- 是否存在失败重试、降级、幂等、状态变化等问题？
- 用户更关心“执行顺序”、还是“职责归属”？

如果信息不完整，AI 可以提出候选 use case，并明确标注为假设。

---

## 五、use-case.md 的推荐结构（强制）

`use-case.md` 必须包含以下四个部分，顺序不可调整。

---

### 1. Use Case Overview（用例概览）

用于列出当前模块最重要的业务动作。

**要求：**
- 使用“动词 + 对象”的方式命名
- 数量应保持克制，只保留真正重要的 use case
- 每个 use case 应能追溯到 `goals-duty.md` 中的职责

**示例（仅作说明）：**
- Build Research Task：构建一次研究任务
- Execute Retrieval Flow：执行检索流程
- Persist Session Memory：写入会话记忆

---

### 2. Main Flow Description（主流程描述）

用于说明每个关键 use case 在当前模块中的执行过程。

**要求：**
- 以“输入 → 判定 / 编排 → 输出”的顺序描述
- 强调步骤的责任归属，而不是实现代码
- 若存在条件分支，应说明触发条件

**推荐描述方式：**
1. 接收输入
2. 校验或标准化
3. 调用内部能力 / 外部依赖
4. 聚合或转换结果
5. 输出结果或产生副作用

**注意：**
- 不要求写类名、函数名
- 不展开为实现级伪代码

---

### 3. Responsibility Boundaries（责任边界）

用于明确当前模块在 use case 执行中的责任落点。

**要求说明：**
- 哪些步骤属于当前模块必须负责
- 哪些步骤由外部模块负责
- 哪些内容属于基础设施、适配器或工具层，而非当前模块核心职责

**该部分用于避免：**
- controller 过胖
- service 过胖
- 编排逻辑四处散落
- 模块边界被流程描述悄悄侵蚀

---

### 4. Failure & Decision Points（失败点与决策点）

用于说明关键 use case 中必须被明确处理的风险节点。

**要求：**
- 至少指出一类失败场景
- 说明当前模块在该场景下的预期行为
- 如存在重试、降级、幂等、跳过、回滚等策略，可在此说明

**示例：**
- 当外部检索为空时，返回空结果并记录原因
- 当模型调用失败时，触发降级摘要策略
- 当重复请求进入时，保持幂等输出

这一部分的作用是让后续实现知道：  
哪些地方是“流程关键点”，不能随意模糊处理。

---

## 六、明确禁止的内容

以下内容**不应出现在 use-case.md 中**：

- 具体类名、函数名、方法签名
- 完整 API 参数定义
- 数据库表结构或 ORM 映射
- 与模块职责无关的完整系统流程
- 过度展开的实现级伪代码

如果描述已经开始接近“怎么编码”，说明文档粒度过细，应主动收敛。

---

## 七、撰写时的原则与约束

### 1. 用例是职责的执行表达，不是需求列表
如果一个 use case 无法映射到明确职责，应考虑删除或回到 `goals-duty.md` 重新讨论。

---

### 2. 编排优先于实现
`use-case.md` 应说明“先做什么、后做什么、由谁负责”，  
而不是说明“调用哪个类、用哪个框架”。

---

### 3. 保持可演进性
当前 use case 描述的是**当前合理流程**，而不是永久固定流程。  
AI 应避免使用绝对化措辞。

---

## 八、完成后的自检清单（AI 必须执行）

在提交 `use-case.md` 前，AI 必须确认：

- 是否每个 use case 都能追溯到 `goals-duty.md` 中的某项职责？
- 是否说明了当前模块真正负责的步骤，而不是把外部行为误写成自身职责？
- 是否存在流程说明过细、已经侵入实现设计的问题？
- 是否至少说明了一类关键失败点或决策点？

如存在问题，应优先精简或回到前置文档讨论。

---

## 九、与其他文档的关系

- `use-case.md` 应建立在 `goals-duty.md` 的职责边界之上
- `architecture.md` 中的结构，应能够支撑 `use-case.md` 中的关键业务动作
- `data-model.md` 中的核心概念，应出现在 use case 的输入、输出或流程说明中
- `dfd-interface.md` 关注模块之间的数据流，`use-case.md` 关注模块内部的职责编排
- `test.md` 中的关键场景，应覆盖 use case 中的主流程与失败点

---

## 十、总结

`use-case.md` 是模块职责落地为业务动作的**编排说明文件**。

它的价值不在于“写得像流程图”，  
而在于：**让后续实现明确知道，当前模块如何在不破坏边界的前提下完成其职责。**
