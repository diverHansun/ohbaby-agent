# database 模块 non-functional.md

本文档定义 `services/database` 模块在功能之外必须满足的工程约束。database 模块是系统的关键路径基础设施，所有业务模块的持久化操作都经过它，因此非功能约束需要在设计阶段显式声明。

---

## 一、Quality Priorities（质量优先级）

按优先级排序：

1. **数据完整性优先于性能**：外键约束、事务原子性、WAL 不可绕过。在两者冲突时，宁可写入慢，不可数据不一致。
2. **故障可见性优先于静默降级**：迁移失败必须阻止启动；`SQLITE_BUSY` 重试耗尽必须抛出明确错误；不允许静默丢数据。
3. **启动时正确性优先于启动速度**：迁移必须在业务模块可用之前完成，即使这意味着 daemon 启动需要额外几百毫秒。
4. **实现简单性优于横向扩展能力**：当前阶段不追求多进程写入、分库、缓存层等复杂机制。单进程单文件在 MVP 场景下已足够。

---

## 二、Operational Constraints（运行约束）

### 并发模型

- **单后端写入口**：daemon 模式下只有 daemon 直接写 SQLite；非 daemon CLI 模式下当前 CLI 进程就是后端入口。不支持多个 ohbaby-agent 进程同时写同一 DB
- **WAL 模式**：允许一个写者与多个读者并发。写操作串行化，读操作（如 UI 查询、scheduler 轮询）不阻塞写
- **同步 API**：`better-sqlite3` 阻塞执行，写操作期间 Node.js 事件循环被短暂阻塞

### 写入延迟约束

- 单条简单 SQL（INSERT/UPDATE）：预期 < 5ms（本地 NVMe/SSD 文件系统）
- WAL 模式下读操作不阻塞写，但写者较多时读者仍可能感受到延迟
- `busy_timeout = 5000ms`：SQLite 层最多等待 5 秒；应用层 retry 附加 jitter，总等待上限设计在数秒内

### 启动时延约束

- `initDatabase()` 会在启动时执行 pending 迁移
- 正常情况（首次启动或少量新迁移）预期完成时间 < 500ms
- 迁移文件累积后启动时间可能增加，但迁移通常一次性执行，后续启动无延迟

### 存储文件约束

- SQLite WAL 模式会产生 `.db-wal` 和 `.db-shm` 伴生文件，不可删除运行中的 WAL 文件
- 不支持 NFS 等网络文件系统（SQLite 文件锁依赖本地文件系统语义）
- 存储路径遵循 XDG 规范，支持 `OHBABY_DB_PATH` 环境变量覆盖

### 资源占用

- 连接数：全进程一个 SQLite 连接实例，不使用连接池
- 内存：无应用层查询缓存，内存占用主要由 SQLite 页缓存决定（默认 ~2MB，可通过 `PRAGMA cache_size` 调整）

## 三、Reliability & Observability（可靠性与可观测性）

### 不可接受的失败

| 场景 | 要求 |
|------|------|
| 迁移部分执行（某步 SQL 成功，某步 SQL 失败） | 不可接受 — 每条迁移必须在事务中，全成功或全回滚 |
| SQLITE_BUSY 导致静默写入丢失 | 不可接受 — 重试耗尽必须抛出 `DatabaseBusyError` |
| 外键约束被绕过 | 不可接受 — `PRAGMA foreign_keys = ON` 必须在每次连接初始化时执行 |
| `getDatabase()` 返回未初始化实例 | 不可接受 — 必须抛出 `DatabaseNotInitializedError` |

### 可接受的降级

| 场景 | 接受的行为 |
|------|----------|
| 高频写入时 `SQLITE_BUSY` | 接受有界重试（带 jitter），超过上限后向上报错 |
| 迁移失败 | 接受后端入口启动失败，阻止进入服务状态 |
| WAL 文件未 checkpoint | 接受进程意外退出时 WAL 文件存在，下次启动时 SQLite 自动恢复 |

### 可观测性要求

- `initDatabase()` 完成时记录结构化日志，包含：运行迁移的版本号列表、耗时
- 迁移失败时记录 `MigrationError`，包含失败版本号和原始 SQL 错误
- `SQLITE_BUSY` 每次重试时记录 warn 日志（含重试次数），耗尽时记录 error
- `DatabaseBusyError` 抛出时应包含足够上下文供调用方诊断

**不要求**：
- 每条 SQL 的 trace 日志（Drizzle 层可选开启，但不要求在生产中默认启用）
- 复杂的指标采集（如 P99 写入延迟）— MVP 阶段不引入可观测性基础设施

## 四、Trade-offs & Deferred Requirements（权衡与暂缓项）

### 暂缓：多进程并发写入支持

**当前不处理**：多个 ohbaby-agent 后端进程同时写同一 SQLite 文件的场景。

**原因**：ohbaby-agent 的目标运行方式是单后端入口。WAL 模式已能处理该入口内部的读写交错，Busy Retry 只作为临时锁冲突的防御性保护。

**如果未来需要**：需要引入进程间锁、或迁移到 PostgreSQL 等支持多进程的数据库。

---

### 暂缓：查询性能优化（连接池、页缓存调优）

**当前不处理**：连接池、内存页缓存大小调优、预编译语句缓存。

**原因**：ohbaby-agent 的查询频率不高（主要在会话开始/消息写入时），单连接同步 API 性能已足够。

**如果未来需要**：在 `ConnectionManager` 中添加 `PRAGMA cache_size` 和 Drizzle 的 prepared statement 缓存。

---

### 暂缓：数据库文件加密

**当前不处理**：SQLite 内容加密（如 SQLCipher）。

**原因**：本地 CLI 工具，数据安全由 OS 文件权限保障，MVP 阶段不引入加密复杂度。

**如果未来需要**：通过存储层适配器替换 `better-sqlite3` 为 SQLCipher 版本。

---

### 暂缓：多数据库文件支持（分库）

**当前不处理**：多租户分库、按会话分文件等场景（N5）。

**原因**：单文件模型已满足 MVP 需求，分库会显著增加连接管理复杂度。
