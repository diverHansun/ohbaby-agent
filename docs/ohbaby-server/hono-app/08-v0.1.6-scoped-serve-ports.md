# 08 · v0.1.6 scoped serve ports

> 本文收束 v0.1.6 的 `ohbaby serve` 发布门：在 web UI 仍是单 project-root 界面时，先解决端口冲突、同 scope 重复启动、不同 scope 并存、清晰输出与本机浏览器打开。完整的单 server 多项目路由仍留在 [`04`](./04-multi-project-runtime.md) 的 Option B。

---

## 1. 发布判断

v0.1.6 不应带着固定 `4096` 的端口冲突发布。web 端已经成为 `ohbaby serve` 的主要消费入口；如果第二个项目启动时直接 `EADDRINUSE`，用户会把它理解成 web UI 不稳定，而不是 daemon 的内部细节。

同时，v0.1.6 也不应提前落完整的 `InstanceStore + x-ohbaby-directory` 多项目宿主。当前 web UI 面向单 project-root；把多项目路由、连接列表、scope 切换和权限归属一起塞进本期，会扩大前后端协议和测试矩阵。

v0.1.6 采用 Option A：

- **一个 foreground `serve` 绑定一个 project-root scope**。
- **同 scope 不重复启动 server**。
- **不同 scope 允许并存，用不同端口**。
- **默认端口可自动避让；显式 `--port` 被占用时失败并解释**。
- **显式 `--host` / `--port` 不被静默改写**：同 scope 已有健康 server 时，只有请求的 host/port 与现有 state 一致才复用；不一致则要求先 `ohbaby serve stop`。

---

## 2. Scope 规则

`scopeRoot` 的解析规则与 [`04`](./04-multi-project-runtime.md) 保持兼容：

1. 以 `ohbaby serve` 启动时的 `workdir` 为输入。
2. 若能找到 git root，则 `scopeRoot = git root`。
3. 若不是 git 目录，则 `scopeRoot = canonical resolved cwd`。

这带来两个用户可见结果：

| 场景 | 行为 |
|------|------|
| 在同一个 repo root 与子目录分别执行 `ohbaby serve` | 视为同一个 scope，不分配新端口 |
| 在两个不同 repo 执行 `ohbaby serve` | 视为不同 scope，允许第二个 server 自动分配可用端口 |
| 在非 git 目录执行 `ohbaby serve` | 以该目录自身为 scope |

scope 状态文件放在 scope 内部的 `.ohbaby/server/` 下，而不是继续使用裸 `.ohbaby/daemon.pid`：

```text
<scopeRoot>/.ohbaby/server/daemon.pid
<scopeRoot>/.ohbaby/server/daemon-state.json
```

状态写入应包含 `scopeRoot`，便于 `serve status`、`serve stop` 和测试确认当前命令操作的是哪个 project-root。

---

## 3. 端口策略

### 默认端口

`ohbaby serve` 默认优先尝试：

```text
127.0.0.1:4096
```

如果该端口可用，就使用 `4096`。

### 自动避让

如果用户没有显式传 `--port`，且 `4096` 已被占用，则 server 改用 OS 分配的可用端口（`port: 0`），并把最终 URL 打印给用户。

示例：

```text
ohbaby web ready: http://127.0.0.1:51243
```

不采用线性自增扫描作为默认策略，避免端口选择与其他进程竞态。测试里只断言第二个 server 得到非 `4096` 的可连接端口。

### 显式端口

如果用户显式传了 `--port`，该端口就是契约。端口被占用时，命令失败并解释：

```text
Port 4096 is already in use for 127.0.0.1.
Choose another port with --port, or stop the process currently using it.
```

显式端口不自动改写，避免脚本和文档引用的地址失真。

## 4. 同 scope 重复启动

启动流程在真正 listen 前先读取当前 scope 的 state：

1. state 不存在：启动新 server。
2. state 存在但 pid 已死：由 pid lock 的 stale takeover 机制清理并启动新 server。
3. state 存在且 pid 存活：用 state 中的 `host`、`port`、`authToken` 调 `/api/health`。
4. health 成功：不启动第二个 server，返回已有 server。
5. health 失败且 pid 仍存活：不强行接管，报错提示用户先 `ohbaby serve stop` 或清理 `<scopeRoot>/.ohbaby/server`。v0.1.6 不做活进程覆盖，避免误杀/误接管。

同 scope 复用时，终端输出仍给出浏览器地址：

```text
ohbaby web ready: http://127.0.0.1:4096
```

这保证用户在同一 project-root 下继续通过同一个 web UI 新建会话，而不是得到另一个 backend。

---

## 5. status / stop 语义

v0.1.6 不做全局 `serve ps`。默认管理命令只作用于当前 scope：

```bash
ohbaby serve status
ohbaby serve stop
```

`status` 读取当前 `scopeRoot/.ohbaby/server/daemon-state.json`。`stop` 只停止当前 scope 对应的 pid，并要求 state 中的 `pidToken` 与同目录 `daemon.pid` 的 token 匹配；若无法证明 ownership，则拒绝 stop，避免 stale state + PID 复用时误杀无关进程。

输出保持简洁：

```text
daemon status: running pid=12345 url=http://127.0.0.1:4096
```

如果当前目录没有 server：

```text
daemon status: not-running
```

## 6. 浏览器打开

`ohbaby serve` 在成功启动或复用 server 后，默认尝试打开最终 URL：

```text
ohbaby web ready: http://127.0.0.1:4096
```

实现边界：

- CLI 层负责打开浏览器，server 层只返回 URL。
- 打开失败不应停止 server；只在 stderr 给一行可解释提示。
- 单元测试通过可注入 `openUrl` 验证调用，不实际打开浏览器。
- 若后续需要脚本友好模式，可增加 `--no-open`，但 v0.1.6 的默认 UX 是自动打开。
- `/api/health` 探测必须有短超时，避免端口 accept 但不响应时阻塞 `ohbaby serve`。

## 7. SQLite 与多 project 并存

当前默认 SQLite 路径是用户级全局库，而不是 per-project 库。v0.1.6 允许不同 project-root 的多个 foreground server 并存后，必须用测试证明：

- project A 与 project B 的 session 列表按 `projectRoot` 过滤，不互相显示。
- project A 的 active session 不会污染 project B 的 fresh startup。
- 两个 server 使用不同端口，但可共享同一个默认 DB。
- 同 session 并发 run 仍由 `run_ledger` owner/claim 保护。

本期不把 DB 改成 per-project。这样保留全局历史、跨项目列表和未来 v0.1.7 `InstanceStore` 的迁移空间。

## 8. 验收门

自动化测试必须覆盖：

1. 同 git root 下两次 `startDaemonServer()` 复用同一个 URL。
2. 不同 git root 在默认端口冲突时第二个 server 自动获得可用端口。
3. 显式 `--port` 被占用时报错，不 fallback。
4. 同 scope 已有健康 server 时，显式改 `--host` / `--port` 会报错，不静默复用旧地址。
5. 活 pid 但 health 失败时报错，不覆盖接管。
6. `serve status` / `serve stop` 操作当前 scope，且 `stop` 校验 pid lock ownership。
7. CLI 成功启动或复用后调用 `openUrl(url)`。
8. SQLite 多 project 并存时 active session 与 session 列表按 project root 隔离。

手工 smoke：

```bash
pnpm build
pnpm --filter ohbaby-cli start serve
```

浏览器应自动打开 URL。另起不同 repo：

```bash
pnpm --filter ohbaby-cli start serve
```

终端应打印另一个可用端口的 URL。同 repo 子目录重复执行时，应打印同一个 URL。

---

## 9. 自审

- 是否解决 v0.1.6 的发布风险？是，固定端口冲突、同 scope 重复 daemon、不同 scope 并存都进入验收门。
- 是否提前实现 v0.1.7 多项目 UI/路由？否，`x-ohbaby-directory`、`InstanceStore`、全局 `serve ps` 仍在 [`04`](./04-multi-project-runtime.md)。
- 是否保留默认 CLI in-process 原则？是，本文只约束显式 `ohbaby serve`。
- 是否解释 SQLite 风险？是，保持用户级 DB，并要求多 project 并存测试证明隔离语义。
- 是否给用户清晰启动体验？是，`ohbaby web ready: <url>` 是唯一主输出，默认打开浏览器。
