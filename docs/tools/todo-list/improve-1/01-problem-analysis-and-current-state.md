# TodoList 现状与问题分析

## 1. 当前代码基线

现有实现已经具备可复用的后端雏形：

- `packages/ohbaby-agent/src/tools/todo.ts` 已注册 `todo_read` 与 `todo_write`。
- `InMemoryTodoStore` 已按 `sessionId` 隔离列表并做防御性复制。
- `packages/ohbaby-agent/src/tools/builtin.ts` 已把两个工具纳入内置工具集合。
- `todo_read` 已标记为只读，调度器层面的读写分类与最终方向一致。
- 单元测试已覆盖基本写读、session 隔离、非法状态与防御性复制。

当前缺口主要不在“有没有工具”，而在领域契约过重、服务生命周期未接入 runtime、恢复/投影缺失以及 Web/TUI 完全未实现。

## 2. 问题清单

### P-01：领域模型与目标契约不一致

**现状**：`TodoItem` 强制 `id`，支持 `priority` 和 `cancelled`；工具提示还要求只能有一个 `in_progress`。

**影响**：Agent 每次整体重写时要维护没有业务价值的身份和优先级字段；工具输入更易出错，文档与代码也会继续分叉。

**根因**：早期实现沿用了通用任务管理模型，没有按“短生命周期 Agent 进度清单”收缩。

### P-02：缺少硬性边界和原子写入契约测试

**现状**：列表数量、内容长度均无限制；没有验证第 11 项、超过 100 字符、空白内容以及部分非法列表不会污染旧状态。

**影响**：模型可能生成过长列表，占据 UI 和上下文；失败边界不可预测。

**根因**：现有校验只关注字段类型，没有把产品约束落到 schema 与执行层。

### P-03：TodoStore 生命周期短于 runtime

**现状**：`createBuiltinTools()` 默认即时创建 `InMemoryTodoStore`；composition 未显式持有稳定 Todo 服务，也没有恢复入口和事件订阅口。

**影响**：无法可靠地把工具写入、session resume、snapshot 和 SSE 事件连接成一条链。

**根因**：现有 Todo 被作为独立工具文件完成，尚未提升为 UI runtime 的 session-scoped 能力。

### P-04：缺少可靠恢复语义

**现状**：代码没有从消息历史恢复；旧文档只写“最后一次工具调用”，没有要求调用成功，也没有区分“未加载”和“已加载空数组”。

**影响**：失败写入可能错误覆盖事实；显式清空后可能在后续读取时复活旧列表。

**根因**：没有把 tool-call 与对应成功 tool-result 作为一个恢复事务看待。

### P-05：SDK、snapshot 和事件没有 Todo 投影

**现状**：`UiSnapshot` 没有 session Todo；`UiEvent` 没有 `todo.updated`；UI state store、Web reducer 和 TUI store 均无相应分支。

**影响**：Web/TUI 如果直接实现只能各自扫描消息历史或形成非正式通道，重连、切换 session 和事件丢失后的状态会不一致。

**根因**：后端工具先于 UI projection 实现。

### P-06：正常 transcript 会暴露 Todo 工具噪音

**现状**：通用消息投影会把 tool call/result 转换为 `UiMessagePart`，Web/TUI 也会通用渲染工具部分。

**影响**：同一进度同时出现在列表面板和对话历史，频繁更新会淹没真正的工作内容。

**根因**：缺少工具级展示策略；持久消息历史和面向用户的 transcript 尚未明确分层。

### P-07：Web 没有 TodoDock

**现状**：Web 有 composer、prompt queue、transcript 和对应 reducer/selectors，但没有 Todo selector、组件或样式。

**影响**：Web 用户看不到结构化进度；如果简单渲染前 5 项，会隐藏真实状态并制造额外分页规则。

**根因**：前端设计尚未进入实现。

### P-08：TUI 没有紧凑选择、展开键与布局

**现状**：TUI 没有 Todo store 字段和组件；Prompt 附近也没有 TodoPanel；`Ctrl+T` 尚未分配给该功能。

**影响**：直接展示 10 项会挤压终端，固定截断又会漏掉正在进行的任务。

**根因**：终端有限高度下的选择策略和键盘交互尚未定义。

### P-09：生命周期边界没有统一事实

**现状**：旧文档把显示/隐藏完全交给各 UI，但仅凭列表和当前 run 状态无法在“本轮刚全部完成”和“上轮已全部完成”之间稳定区分，重连时尤其明显。

**影响**：Web/TUI 可能在 run 结束后仍显示，或在当前 run 全部完成时过早消失。

**根因**：领域事实与展示投影没有分层，也没有由后端统一投影可见性。

### P-10：测试策略与目标交付不匹配

**现状**：旧文档明确排除 Web/TUI 和浏览器 E2E；仓库虽有 Vitest、Web reducer/组件测试、Ink TUI 测试和真实 CLI 进程样板，但 Todo 尚未覆盖这些层级。

**影响**：可能出现类型通过但真实 UI 不显示、键位无效、transcript 泄漏或重连不同步。

**根因**：旧测试范围停留在工具内部，没有覆盖用户可见行为。

## 3. 约束与风险

| 风险 | 约束/缓解 |
|------|-----------|
| 无 `id` 导致 UI key 不稳定 | key 只作为渲染细节，使用 index 与内容组合；不得为解决 key 反向扩展领域模型 |
| 隐藏 transcript 后影响恢复 | 只在 UI 投影/渲染层隐藏，底层消息与工具结果仍保留 |
| session 恢复扫描成本 | 仅首次加载时从后向前扫描，找到最后一次成功写入即停止，并缓存“已加载”状态 |
| 多 `in_progress` 挤满 TUI | 紧凑态最多取前 5 个进行中项目并保留原数组顺序；展开态显示全部 |
| snapshot 向后兼容 | Todo 投影字段采用可选字段，旧 snapshot 视为空列表 |
| 重复事件引起闪烁 | 成功写入后先做小数组结构比较，相同则不发布更新事件 |

## 4. 结论

不需要推翻现有工具体系。正确改造路径是：先收缩 TodoItem 和校验契约，再把 TodoStore 提升为 runtime 持有的服务，补齐恢复与 SDK 投影，最后分别实现 Web/TUI 只读视图和真实端到端验收。
