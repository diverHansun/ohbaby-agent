# Layouts 布局层概述

## 一、定位

布局层（Layouts）负责定义终端屏幕的区域划分和空间分配。它决定"内容放在哪里"，而不关心"内容是什么"。

布局层与视图层（Views）构成上下层关系：布局提供容器结构，视图填充具体内容。这种分离确保屏幕结构变更不影响业务视图，业务视图变更也不影响整体布局。

## 二、布局列表

当前仅有一个布局组件：

| 布局 | 文档 | 职责 |
|------|------|------|
| DefaultLayout | [default-layout.md](./default-layout.md) | 三区域固定底部布局 |

## 三、设计原则

### 3.1 布局与内容分离

布局组件通过 `children` 接收视图内容，不导入也不引用任何具体的 View 组件。这意味着布局组件可以包裹任意视图而无需修改自身代码。

### 3.2 固定底部模式

所有布局遵循"固定底部"原则：Prompt 和 StatusBar 始终钉在终端窗口底部，不随内容量变化而移动。用户视线和输入焦点始终锁定在底部区域。

### 3.3 最小 Context 依赖

布局组件仅读取渲染自身所需的最少 Context 数据。DefaultLayout 仅读取 `AppStateContext.loading` 用于条件渲染加载指示器，不读取消息、会话、配置等数据。

### 3.4 可扩展预留

当前仅有 DefaultLayout。如果未来需要其他布局形态（如无障碍布局、精简布局），可以新增布局组件并在 App 层切换，而不影响视图层代码。

## 四、布局与视图的组装关系

```
App.tsx
  └── Provider 树
        └── DefaultLayout          ← 布局：决定区域划分
              ├── {children}       ← 视图：由 Router 输出
              ├── LoadingIndicator ← 布局内组件：条件渲染
              ├── Prompt           ← 布局内组件：固定
              └── StatusBar        ← 布局内组件：固定
```

视图通过 Router 确定后，作为 `children` 传入布局组件。布局组件不感知当前渲染的是哪个视图。
