# Agent Loop 长期运行修补与优化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让主 agent 和子代理在真实编码任务中稳定长跑：同步子代理短而快，异步子代理可长期运行，provider 网络抖动可自动恢复，`maxSteps` 不再造成过早失败。

**Architecture:** 将长期运行控制拆成四层：lifecycle step safety cap、subagent wall-clock deadline、tool-level timeout、provider retry。主 loop 负责 step 和最终文本收束；同步/异步子代理分别在 AgentService 和 AgentTaskManager 中施加 deadline；LLM retry 独立封装在 llm-client 层。

**Tech Stack:** TypeScript, Vitest, existing ohbaby-agent lifecycle/run-manager/tool-scheduler/agents architecture.

## 2.1 最终默认值建议

| 配置项 | 默认值 | 硬约束 | 说明 |
|---|---:|---:|---|
| 主 agent `maxSteps` | `1000` | 可配置 | 借鉴 Kimi-code 的宽松上限；防死循环，不作为普通任务预算 |
| `build` agent `maxSteps` | `1000` | 可配置 | 复杂编码任务默认主入口 |
| `plan` agent `maxSteps` | `1000` | 可配置 | 计划任务也属于主 agent；简单问题会自然提前结束 |
| `explore` subagent `maxSteps` | `50` | 可配置 | 同步探索由 5min deadline 主导 |
| `generic/research` subagent `maxSteps` | `100` | 可配置 | 给短任务足够空间 |
| 同步 `task` service deadline | `300_000ms` | 不超过 `300_000ms` | AgentService 是 5min 语义的权威来源 |
| scheduler `task` guard timeout | `310_000ms` | 可配置 | 仅防止工具永久挂住，给 service deadline 10s cleanup grace |
| 异步 background task timeout | `1_800_000ms` | 可配置到 `3_600_000ms` | 默认 30min，最高 1h |
| provider `maxRetriesPerStep` | `5` | 用户可配 `0-10` | 表示首次失败后最多再试 5 次 |
| provider retry backoff | `500ms -> 10s` | 尊重 `Retry-After`，单次 sleep cap `60s` | 指数退避 + jitter |

deadline controller 必须在 `AgentService` 中创建并传给 `runAgent()`，而不是依赖 scheduler 抢先 abort：

```typescript
function createDeadlineController(input: {
  readonly parent?: AbortSignal;
  readonly timeoutMs: number;
  readonly reason: string;
}): { readonly signal: AbortSignal; readonly dispose: () => void } {
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(input.reason), input.timeoutMs);
  const abortFromParent = (): void => controller.abort(input.parent?.reason);
  input.parent?.addEventListener("abort", abortFromParent, { once: true });

  return {
    signal: controller.signal,
    dispose: () => {
      clearTimeout(timeout);
      input.parent?.removeEventListener("abort", abortFromParent);
    },
  };
}
```

- [ ] **Step 3: 实现 backoff 和 `Retry-After`**

```typescript
export function nextRetryDelayMs(input: {
  readonly attempt: number;
  readonly error: unknown;
  readonly policy: ProviderRetryPolicy;
  readonly random: () => number;
}): number {
  const retryAfter = parseRetryAfterMs(input.error);
  if (retryAfter !== undefined) {
    return Math.min(retryAfter, input.policy.retryAfterCapMs);
  }

  const base = Math.min(
    input.policy.initialDelayMs * 2 ** Math.max(0, input.attempt - 1),
    input.policy.maxDelayMs,
  );
  const jitter = 0.8 + input.random() * 0.4;
  return Math.floor(base * jitter);
}
```

## 2.12 已确认决策

以下是推荐默认值；如果用户无特别意见，实施时按这些值执行：

| 决策 | 推荐默认 | 原因 |
|---|---|---|
| scheduler `task` guard | `310_000ms` | `AgentService` 5min deadline 后留 10s 清理和写结果 |
| provider retry 语义 | `maxRetriesPerStep = 5` | 表示首次失败后最多再试 5 次，符合“默认重试 5 次”直觉 |
| 主 agent 默认 maxSteps | `1000` | 直接借鉴 Kimi-code；简单问题会自然提前停止，不会因上限提高而变慢 |
| `StreamStopReason` 命名 | `provider_finished` / `user_aborted` | stream response 只描述已 yield 的本地停止原因；provider 中断走结构化 error/lifecycle terminal reason |
| `AgentTerminalReason` 范围 | 只覆盖第一阶段原因 | 避免为了完美枚举提前设计未发生的终态 |
