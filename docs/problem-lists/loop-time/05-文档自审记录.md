## 5.2 已采纳的复核意见

| 复核意见 | 处理方式 |
|---|---|
| `lifecycle.ts` / `scheduler.ts` 体量过大 | 记录为风险；本轮只新增小 helper，不做大拆分 |
| scheduler 的 `120_000ms` 控制同步子代理属于控制耦合 | 将 `AgentService` 定为同步子代理 deadline 权威；scheduler 只保留 guard |
| `AgentConfig.timeout` 类型存在但无效 | 计划中明确同步/异步两条消费路径和配置边界 |
| `finishReason: "length"` / `"(Interrupted)"` 是哨兵值泄漏 | 新增 `StreamStopReason` 计划，不再伪装 provider finish reason |
| provider retry v1 不应做 mid-stream 重放 | 保留“未产生 delta 才自动 retry”的保守策略 |
| terminal reason 需要结构化 | 增加第一阶段最小 `AgentTerminalReason` 联合类型 |
| retry 过程不能像卡住 | `llm:retrying` 透传到 `run.llm.retrying`，并由 adapter event source 还原 |
| 时间测试不能真实等待 | 测试文档要求 fake timers 或可注入 clock |

## 5.3 明确不纳入本轮的反馈

以下反馈成立，但会扩大范围，不纳入 loop-time 第一阶段：

- 不拆 `scheduler.ts` 闭包工厂。
- 不迁移 `PermissionPort` 出 tool-scheduler。
- 不清理 `AgentTaskStatus` 中暂未使用的 `idle/blocked`。
- 不做 `agent-task.ts` 4 个 tool 的 generic factory 重构。
- 不处理 `confirmExternalReadPermission` / `confirmExternalWritePermission` 的 DRY 问题。
- 不实现 Claude Code 风格 persistent retry。

## 5.4 关键一致性检查

| 项 | 当前文档一致性 |
|---|---|
| 同步子代理 deadline | `AgentService` 权威，默认 `300_000ms`，运行时不超过 5min |
| scheduler guard | `task = 310_000ms`，`agent_open` 保持默认 `120_000ms` |
| 异步子代理 timeout | 默认 `1_800_000ms`，可配置到 `3_600_000ms` |
| provider retry | `maxRetriesPerStep = 5`，用户 abort 不 retry |
| stream reason | provider `finishReason` 与本地 `streamStopReason` 分离 |
| maxSteps | 主 agent 默认 `1000`；到达上限时文本收束 |
| terminal reason | 第一阶段只覆盖 lifecycle result 实际新增终态；run ledger 持久化放后续 |

## 5.5 已确认决策项

当前没有阻塞文档修订的决策。以下值已按本轮决策固化到计划中：

- scheduler `task` guard 采用 `310_000ms`。
- provider retry 采用“首次失败后最多再试 5 次”的语义。
- 主 agent 默认 `maxSteps` 采用 Kimi-code 风格的 `1000`。
- `StreamStopReason` 只覆盖已 yield response 的 `provider_finished` / `user_aborted`；provider 中断由 lifecycle `terminalReason` 表达。

## 5.6 自审结论

文档已从“长期运行方向计划”收敛为更可实施的分层方案：

- agent/service 层拥有同步子代理 deadline；
- task manager 层拥有异步子代理 deadline；
- scheduler 层只拥有工具 guard timeout；
- llm-client 层拥有 provider retry 和 stream stop reason；
- lifecycle/run-manager 层消费结构化结果、terminal reason 和 retry 可观察事件。

这能避免把所有超时与 retry 逻辑继续堆进 `lifecycle.ts` 或 `scheduler.ts`，同时保留第一阶段的实现可控性。
