# 3. claude-code / kimi-code / opencode 借鉴点

## 3.1 参考结论摘要

| 项目 | maxSteps/maxTurns 策略 | 子代理/后台任务策略 | provider retry 策略 | 对 ohbaby 的启发 |
|---|---|---|---|---|
| Kimi-code | 默认 `max_steps_per_turn = 1000`，硬上限 | 有 `agent_task_timeout_s` 后台任务 timeout 配置 | `chatWithRetry` 默认 3 次，retry event 可观察 | `maxSteps` 应宽松；retry 应在 loop step 层可观察 |
| opencode | `steps` 可选，默认接近无限；最后一步注入 MAX_STEPS 文本收束 | 不把 steps 当普通失败 | retry policy 尊重 `retry-after-ms` / `retry-after` | 最后一步文本收束体验最好 |
| claude-code | `maxTurns` 可选；达到后发 `max_turns_reached` attachment | async agent 独立；fork subagent 默认 `maxTurns: 200`；多代理探索 30min timeout | API `withRetry` 默认较强，错误分类细 | 区分同步/异步；终态要结构化；复杂多代理任务可 30min |

## 3.2 Kimi-code

参考文件：

- `D:\Projects\Code-cli\kimi-code\packages\agent-core\src\loop\run-turn.ts`
- `D:\Projects\Code-cli\kimi-code\packages\agent-core\src\loop\retry.ts`
- `D:\Projects\Code-cli\kimi-code\packages\agent-core\src\agent\turn\index.ts`
- `D:\Projects\Code-cli\kimi-code\packages\agent-core\src\session\index.ts`
- `D:\Projects\Code-cli\kimi-code\docs\zh\configuration\config-files.md`

观察点：

- `DEFAULT_MAX_STEPS = 1000`，文档默认 `max_steps_per_turn = 1000`。
- `max_retries_per_step = 3`，通过 `chatWithRetry()` 包住单步 LLM call。
- retry 会发 `step.retrying` 事件，包含 attempt、delay、error 信息。
- 配置 schema 中有 `agentTaskTimeoutS`。
- session 层读取 `config.background?.agentTaskTimeoutS` 并转换为毫秒。

可借鉴：

- 主 loop 的默认上限应该足够宽松，避免正常任务被误杀。
- retry 应该是 step-level，而不是散落在 provider adapter。
- retry 过程必须可观察，否则用户会误以为 agent 卡住。
- 后台 agent task timeout 是独立配置，不应依赖普通工具 timeout。

不直接照搬：

- Kimi 的 `1000` 适合作为主 agent 默认上限：它是安全阀而不是预算；简单问题会在模型自然完成时结束，不会因为上限更高而被刻意拉长。
- Kimi maxSteps 是硬 error；ohbaby 更适合采用 opencode 的最后一步文本收束。

## 3.3 opencode

参考文件：

- `D:\Projects\Code-cli\opencode\packages\opencode\src\config\agent.ts`
- `D:\Projects\Code-cli\opencode\packages\opencode\src\session\prompt.ts`
- `D:\Projects\Code-cli\opencode\packages\opencode\src\session\prompt\max-steps.txt`
- `D:\Projects\Code-cli\opencode\packages\opencode\src\session\retry.ts`
- `D:\Projects\Code-cli\opencode\packages\opencode\src\session\processor.ts`

观察点：

- agent config 使用 `steps`，`maxSteps` 已标记 deprecated。
- `steps` 的说明是“Maximum number of agentic iterations before forcing text-only response”。
- `agent.steps ?? Infinity`，即默认不强制低步数。
- 到最后一步时，opencode 将 `MAX_STEPS` 注入消息，要求工具禁用并输出文本。
- retry policy 会解析 `retry-after-ms` 和 `retry-after`。
- processor 层使用 retry policy，并把 retry 状态暴露给 session。

可借鉴：

- `maxSteps` 到达时不应直接失败，应强制模型做最终文本总结。
- “工具禁用直到下一次用户输入”的语义非常适合减少用户困惑。
- `Retry-After` 应优先于本地 backoff。

不直接照搬：

- opencode 默认 `Infinity` 对 ohbaby 第一阶段过于开放；需要保留一个较高但有限的安全阀。
- opencode 的 Effect retry 体系不需要迁移；ohbaby 用普通 async helper 即可。

## 3.4 claude-code

参考文件：

- `D:\Projects\Code-cli\claude-code\src\query.ts`
- `D:\Projects\Code-cli\claude-code\src\QueryEngine.ts`
- `D:\Projects\Code-cli\claude-code\src\services\api\withRetry.ts`
- `D:\Projects\Code-cli\claude-code\packages\builtin-tools\src\tools\AgentTool\runAgent.ts`
- `D:\Projects\Code-cli\claude-code\packages\builtin-tools\src\tools\AgentTool\forkSubagent.ts`
- `D:\Projects\Code-cli\claude-code\src\commands\ultraplan.tsx`

观察点：

- `query.ts` 支持可选 `maxTurns`，达到后发 `max_turns_reached` attachment 并返回 `reason: "max_turns"`。
- AgentTool 的 agent frontmatter 支持 `maxTurns`。
- `forkSubagent.ts` 中 fork agent 默认 `maxTurns: 200`。
- `runAgent.ts` 明确区分 `isAsync`，并传递 `maxTurns: maxTurns ?? agentDefinition.maxTurns`。
- `ultraplan` 注释说明多代理探索较慢，默认 30min timeout。
- `withRetry.ts` 默认重试能力强，处理连接错误、408/409/429/5xx、overloaded/529，并支持 retry event。

可借鉴：

- max-turns/max-steps 终态应结构化，不能只给普通字符串。
- async 子代理应作为长期任务处理，而不是受父请求短超时支配。
- 多代理探索类任务默认 30min 是合理量级。
- provider retry 分类要覆盖连接错误、rate limit、server overload。

不直接照搬：

- Claude Code 的 retry 机制很重，包含持久 retry、客户端刷新、认证恢复等；ohbaby 第一阶段只需要 provider 网络/API retry。
- 不建议一开始引入“无限 persistent retry”，否则用户可能难以判断任务是否真的卡住。

## 3.5 对 ohbaby 的统一借鉴结论

1. `maxSteps` 应是高安全阀，默认不该过低。
2. 到达 `maxSteps` 时优先文本收束，而不是直接失败。
3. 同步子代理和异步子代理的运行时间策略必须分开。
4. provider retry 要集中实现，不能分散在各 provider adapter。
5. retry、maxSteps、timeout 都要有事件和结构化 terminal reason。
6. 后台任务 timeout 属于任务管理器，不属于 `agent_open` 工具调用本身。
7. 工具调度器可以提供 timeout guard，但不应拥有同步/异步 agent deadline 的产品语义。
8. provider finish reason 与本地 abort/interrupt reason 应分开表达，避免上层误判。
