# 1. 现有问题和涉及到的模块

## 1.1 背景结论

本轮关注的是 agent loop 长期运行能力：主 agent 和子代理不应因为过小的步数上限、普通工具超时、一次 provider 网络抖动而过早结束。工具调用本身可以失败并由模型再次调用；真正需要避免的是 lifecycle/run 被自动 abort，导致用户不得不反复补 prompt。

本轮修补前，ohbaby-agent 的停止路径主要来自四类机制：

| 停止来源 | 修补前行为 | 用户侧风险 |
|---|---|---|
| lifecycle `maxSteps` | 修补前 `DEFAULT_MAX_STEPS = 8`，内置 agent 覆盖为 15/30/50；本轮已确认主 agent/build/plan 维持 `1000` | 复杂任务可能在正常探索、修改、验证链路中提前结束 |
| 外部 `AbortSignal` | run-manager、subagent runner、tool scheduler 都会传递 abort | 用户主动取消时正确；但工具超时也可能级联取消同步子代理 |
| tool scheduler timeout | 默认 `120_000ms` 包住所有工具，包括 `task` | 同步子代理作为工具运行，超过 2min 会被当作工具超时取消 |
| provider/API error | 非 context overflow 的 provider 错误直接抛出 | 网络抖动、429、5xx、连接失败没有统一重试，容易终止 run |

这说明当前系统缺少一套分层的长期运行策略。`maxSteps`、tool timeout、subagent deadline、provider retry 应当分工明确，而不是互相替代。

## 1.2 修补前源码观察与当前决策

### lifecycle 主循环

核心文件：

- `packages/ohbaby-agent/src/core/lifecycle/lifecycle.ts`
- `packages/ohbaby-agent/src/core/lifecycle/types.ts`

关键现状：

- 修补前 `DEFAULT_MAX_STEPS = 8`；当前决策为主 agent 默认 `1000`，内置 `build` / `plan` 也维持 `1000`。
- `LifecycleSessionParams.maxSteps` 可覆盖默认值，但没有 wall-clock deadline。
- 主循环使用 `for (let step = 1; step <= maxSteps; step += 1)`。
- 修补前旧实现中，当最后一步仍产生工具调用时会直接返回失败结果：`finalResponse: "Maximum lifecycle tool steps reached"`。
- 修补前 LLM 调用只传入 `signal`，没有 provider retry policy。

问题：

- `maxSteps` 当前是硬失败阀，而不是最后一步文本收束阀。
- 默认步数太低，和真实编码任务的自然循环不匹配。
- 生命周期没有“主 agent 可长跑，子代理按模式限时”的配置入口。

### run-manager 与 runner

核心文件：

- `packages/ohbaby-agent/src/runtime/run-manager/manager.ts`
- `packages/ohbaby-agent/src/runtime/run-manager/worker.ts`
- `packages/ohbaby-agent/src/core/agents/runner.ts`

关键现状：

- `RunManager` 为每个 run 创建 `AbortController`。
- `RunWorker` 将 `abortSignal` 和 `maxSteps` 传入 lifecycle。
- `runAgent()` 会把外部 signal 绑定到 run cancellation。

问题：

- run cancellation 能传递，但没有区分用户取消、同步子代理 deadline、异步子代理 deadline、provider exhausted retry。
- 终态原因不够结构化，后续 UI 或 resume 逻辑难以判断“是否可恢复”。

### 同步子代理 `task`

核心文件：

- `packages/ohbaby-agent/src/tools/task.ts`
- `packages/ohbaby-agent/src/agents/service.ts`
- `packages/ohbaby-agent/src/agents/types.ts`

关键现状：

- `task` 是普通 builtin tool，`category: "subagent"`。
- `createTaskTool()` 将 `context.signal` 传给 `executor.execute()`。
- `AgentService.executeTask()` 使用 `waitMode: "waitForCompletion"` 等待子 run 完成。
- `AgentConfig.timeout` 已存在于类型和配置 schema，但运行时没有消费。

问题：

- 同步子代理会被 tool scheduler 的 `120_000ms` 默认超时包住。
- 用户期望同步子代理短而快，但 2min 对代码探索、局部修复、验证来说偏紧。
- `timeout` 字段存在但无效，配置表达和实际运行行为不一致。

### 异步子代理 `agent_open`

核心文件：

- `packages/ohbaby-agent/src/tools/agent-task.ts`
- `packages/ohbaby-agent/src/agents/tasks/manager.ts`
- `packages/ohbaby-agent/src/agents/tasks/types.ts`

关键现状：

- `agent_open` 只负责打开后台任务，后台 run 由 `AgentTaskManager` 自己创建 `AbortController`。
- 后台任务运行期间不再受父 run 完成影响。
- `agent_close` 会取消当前 active child run。
- `AgentTaskStatus` 目前包含 `pending/running/idle/completed/failed/cancelled/blocked`，没有 `timed_out`。

问题：

- 后台任务适合长期执行，但当前没有 30min/60min 级别 deadline。
- 没有明确的超时终态，用户和父 agent 只能看到普通 failed 或持续 running。

### tool scheduler timeout

核心文件：

- `packages/ohbaby-agent/src/core/tool-scheduler/constants.ts`
- `packages/ohbaby-agent/src/core/tool-scheduler/types.ts`
- `packages/ohbaby-agent/src/core/tool-scheduler/scheduler.ts`

关键现状：

- 默认工具超时是 `120_000ms`。
- scheduler 在超时时 abort 工具执行 controller。
- `task`、`agent_open` 都被归到 `subagent` 类别。

问题：

- 用统一工具 timeout 包所有工具，无法表达 `task` 应为 5min、`agent_open` 只应快速启动、普通工具仍可 120s。
- 如果只把 `subagent` 类别整体加长，会错误扩大 `agent_status/agent_close/agent_eval` 的等待窗口。
- scheduler 不应成为同步子代理 5min 语义的权威来源。同步子代理 deadline 属于 agent/service 层；scheduler 只能作为工具执行 guard，避免工具永久挂住。

### LLM provider streaming

核心文件：

- `packages/ohbaby-agent/src/core/llm-client/streaming.ts`
- `packages/ohbaby-agent/src/services/interface-providers/openai-compatible.ts`
- `packages/ohbaby-agent/src/services/interface-providers/anthropic.ts`

关键现状：

- `streamChatCompletion()` 负责累积 streaming delta 和 tool call delta。
- provider abort 会被转换为 partial response。
- 当前 abort partial 使用 `finishReason: "length"` 和 `"(Interrupted)"` 这类哨兵值表达本地中断。
- 网络错误、429、5xx 等非 abort/provider API 错误直接抛出。
- provider adapter 只传递 `signal`，没有统一 retry wrapper。

问题：

- 网络不稳定会直接终止 run。
- streaming 中途失败如果盲目 retry，可能造成重复 assistant 文本或重复 tool call，因此 retry 必须和“是否已经产生可见 delta”绑定。
- provider 的 finish reason 与本地 stream stop reason 混用，会让 lifecycle 层误判 `"length"` 是模型真实输出上限，而不是用户中断。

## 1.3 用户使用时最可能阻塞或失败的行为

| 行为 | 当前风险 | 期望策略 |
|---|---|---|
| provider 连接不上、慢首包、429/5xx | 当前无统一重试，run 直接失败 | 默认重试 5 次；耗尽后结构化失败或暂停 |
| provider mid-stream 断开 | 可能已有 partial assistant 内容 | 不盲目重放；保留 partial，标记可恢复 |
| 同步 `task` 探索复杂代码 | 2min tool timeout 取消子 run | 同步子代理默认 5min，上限不超过 5min |
| 异步 `agent_open` 后长任务 | 无运行 deadline | 后台 turn 默认 30min，可配置到 60min |
| 主 agent 正常多轮修改验证 | `maxSteps` 太低导致提前失败 | 主 loop 默认提高到 1000；最后一步强制文本总结 |
| 长时间 bash/test/build | 普通工具可能 timeout | 工具可失败并由 loop 决定下一步，不应 abort 整个 lifecycle |
| MCP/外部工具挂住 | scheduler 会超时 | 保持工具级失败，不扩大为 run 级失败 |

## 1.4 模块边界

本次优化应按以下边界拆分，避免把所有逻辑塞进 lifecycle：

| 边界 | 负责内容 | 不负责内容 |
|---|---|---|
| lifecycle | step 上限、最后一步文本收束、provider retry 事件透传 | 子代理任务存储、后台任务状态 |
| llm-client | provider 错误分类、retry/backoff、streaming 安全策略 | 决定任务是否继续执行 |
| tool-scheduler | 工具级 timeout、per-tool timeout override | agent run deadline |
| AgentService | 同步子代理 5min deadline、`AgentConfig.timeout` 消费 | 后台任务调度 |
| AgentTaskManager | 异步子代理 30min/60min deadline、`timed_out` 状态 | 父 run 的 step 控制 |
| run-manager | abort reason 传递、run 终态归档 | provider 错误分类细节 |
| config/agents | 用户可配置的 maxSteps/timeout/retry 字段校验 | 默认运行策略执行 |

## 1.5 设计原则

- `maxSteps` 是防死循环和成本保险，不是正常任务预算。
- 同步子代理用短 deadline 控制，异步子代理用长 deadline 控制。
- 工具 timeout 只让单个工具失败，不应自动让主 loop 过早终止。
- provider 网络问题先自救，默认重试 5 次，仍不可达再返回结构化终态。
- 用户主动取消必须立即生效，不能被 retry 或 deadline 包装吞掉。
- 所有超时/上限/重试耗尽都要有可观察的 reason，方便 UI、日志、测试和 resume。
- 本轮不做大规模 god class 拆分。新增逻辑优先放到小型 helper 或独立文件中，避免继续膨胀 `lifecycle.ts` 和 `scheduler.ts`。
