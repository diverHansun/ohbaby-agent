# 4. 测试与验收方案

## 4.1 测试原则

本轮验证重点是 backend command 输出契约，而不是 TUI 交互体验。

测试原则：

- 优先验证 command service 输出结构，而不是只验证 TUI 文本。
- 保留旧字段兼容性，例如 `/status` 的 `model`、`models`、`status`。
- 对 provider 缺失和 provider 抛错做优雅降级验证。
- 对 `/mcps` 做白名单输出验证，确保不泄露 tools、toolCount、error。
- 对 `/skills.scope` 做运行时边界验证，确保只允许 `user | project`。
- 测试中避免脆弱写法，例如直接依赖 `events[0]`、长 mega-test、`JSON.stringify` 负断言。

## 4.2 Agent 端单元测试

### `/help`

验证点：

- 输出 `subject: "help"`。
- `data.commands` 仍存在。
- `data.categories` 新增并按 category 分组。
- category 内保留原始 `UiCommandSpec` 字段。
- 动态 skill command 会出现在 `skill` category。

核心断言形状：

```typescript
expect(output).toMatchObject({
  kind: "data",
  subject: "help",
  data: {
    commands: expect.any(Array),
    categories: expect.arrayContaining([
      expect.objectContaining({
        name: "system",
        commands: expect.any(Array),
      }),
    ]),
  },
});
```

### `/status`

验证点：

- 保留旧字段：`model`、`models`、`status`。
- 新增字段：`sessionId`、`tools`、`skillsCount`、`mcps`、`context`、`projectRoot`。
- `tools` 按精确 source 计数：`builtin`、`module`、`skill`、`mcp`。
- `mcps` 是 MCP server 状态汇总，不是 MCP tool 列表。
- provider 缺失时不崩溃。
- MCP provider 抛错时不崩溃，MCP 汇总为空。

核心断言形状：

```typescript
expect(output.data).toMatchObject({
  sessionId: "session_1",
  tools: {
    builtin: 1,
    module: 1,
    skill: 1,
    mcp: 1,
  },
  skillsCount: 2,
  mcps: {
    total: 4,
    connected: 1,
    failed: 1,
    disabled: 1,
    disconnected: 1,
  },
  projectRoot: "D:/Projects/app",
});
```

### `/mcps`

验证点：

- `/mcps` 输出 `subject: "mcps"`。
- 输出字段是 `{ servers }`。
- 每个 server 只包含 `name` 和 `status`。
- provider 额外返回 `toolCount` 或 `error` 时，command 层会剥掉。
- provider 缺失或抛错时返回 `{ servers: [] }`。

核心断言形状：

```typescript
expect(output).toMatchObject({
  kind: "data",
  subject: "mcps",
  data: {
    servers: [
      { name: "github", status: "connected" },
      { name: "bad", status: "failed" },
    ],
  },
});
```

### `/skills`

验证点：

- `/skills` 输出 `subject: "skills"`。
- 每个 skill 输出 `name`、`description`、`path`、`commandId`、`scope`。
- `source` 可选，存在时保留。
- 不输出 `pluginId`。
- provider 缺失时返回 `{ skills: [] }`。
- 非法 `scope` 的 skill 不出现在 `/skills`，也不出现在 `/help` 的动态 command 中。

核心断言形状：

```typescript
expect(output).toMatchObject({
  kind: "data",
  subject: "skills",
  data: {
    skills: [
      {
        commandId: "skill.review",
        description: "Review code",
        name: "review",
        path: ["review"],
        scope: "project",
        source: "project-native",
      },
    ],
  },
});
```

## 4.3 Catalog 测试

验证点：

- builtin catalog 包含 `mcps` 和 `skills`。
- `/mcps` alias `/mcp` 可解析。
- `/skills` 无 plugin 相关字段。
- 两个命令都暴露在 `tui`、`stdout`、`headless` surface。

核心断言：

```typescript
expect(ids).toEqual(expect.arrayContaining(["mcps", "skills"]));
expect(mcps?.aliases).toContainEqual(["mcp"]);
```

## 4.4 Runtime 和 Adapter 测试

验证点：

- `UiRuntimeComposition.listMcpServerSummaries()` 从 `McpManager.getStatus()` 读取 server 状态。
- runtime summary 只返回 `{ name, status }`。
- `UiRuntimeComposition.getContextUsage()` 使用 context manager 组装当前 session usage。
- `ui-inprocess` command service 能执行 `/skills` 并保留 project skill 的 `scope/source`。
- command catalog 通过 in-process backend 暴露 `mcps` 和 `skills`。

## 4.5 TUI Store 测试

TUI 本轮只做最小文本格式化，测试集中在 `packages/ohbaby-cli/src/tui/store/events.unit.test.ts`。

验证点：

- `/status` notice 可读展示 status/model/session/tools/skills/mcps/context/project。
- `/help` notice 优先展示 categories。
- `/mcps` notice 只展示 server name/status，不展示 toolCount/error。
- `/skills` notice 展示 name/scope/source/description。
- 长 command output 仍按现有 notice 限制截断。

测试质量要求：

- 不依赖 `events[0]` 这种脆弱索引。
- 不把多个不相关 subject 塞进一个超长 mega-test。
- 负断言尽量检查结构化字段，不用 `JSON.stringify(...).not.toContain(...)`。

## 4.6 真实验证

使用根目录 `.env` 中的真实模型 key，通过现有脚本加载到当前进程：

```powershell
pnpm run test:smoke:real
pnpm run test:e2e:snapshot
pnpm run test:e2e:snapshot
pnpm run test:e2e:snapshot
```

说明：

- `test:smoke:real` 会加载 `.env`，要求 `ZAI_API_KEY` 或 `ZHIPU_API_KEY`。
- `test:e2e:snapshot` 连跑多轮，用来观察真实 API-backed E2E 是否稳定。
- 不能打印 API key。

## 4.7 子代理检查

完成实现后使用子代理做只读审查，重点检查：

- `/mcps` 是否泄露 tools、toolCount、error。
- `/skills.scope` 是否只靠 TypeScript 而缺少运行时边界。
- provider 抛错是否会导致 `/status` 或 `/mcps` 整体失败。
- 文档是否仍保留早期 `/mcps` 工具列表设计。

子代理提出的问题必须复核后再修，不盲目照单修改。
