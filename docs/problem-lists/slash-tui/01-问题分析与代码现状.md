# 1. 问题分析与代码现状

## 1.1 问题总览

Slash-commands 系统整体架构（SDK → Agent → CLI/TUI）已搭建完成，命令的解析、解析、分发、事件总线链路通畅。但存在四个具体缺口：

| 命令 | 状态 | 核心问题 |
|------|------|----------|
| `/help` | 已实现 | TUI 端 `formatDataCommandOutput()` 对 subject `"help"` 无特殊处理，走 `JSON.stringify` 通用分支，输出不可读 |
| `/status` | 已实现 | 信息量太少——仅显示 `model.name` + `"idle"/"running"`；`models` 列表、session 信息等均未利用 |
| `/mcps` | **不存在** | 无命令定义、无 handler；`CommandToolProvider` 已在 adapter 接线但无 handler 消费 |
| `/skills` | **不存在（列表面）** | 技能已注册为独立动态命令（`/skillname`），但缺少 `/skills` 列出所有可用技能 |

## 1.3 `/help` 现状

### 命令定义 (catalog.ts)
```ts
{ id: "help", path: ["help"], aliases: [["?"]], category: "system",
  description: "List available commands", surfaces: ["tui", "stdout", "headless"] }
```

### Handler (builtin.ts:120-130)
handler 是 standalone function，在 `createBuiltinHandlers()` 中包装为 `CommandHandler` 对象：
```ts
async function handleHelp(
  helpers: BuiltinHandlerHelpers,
  context: CommandRunContext,
): Promise<void> {
  const catalog = await helpers.listCommands?.(context.surface);
  context.emitOutput(
    dataOutput("help", {
      commands: catalog?.commands ?? [],
    }),
  );
}
```
注册方式（builtin.ts:399-404）：
```ts
{
  id: "help",
  async execute(_invocation, context): Promise<void> {
    return handleHelp(helpers, context);
  },
},
```
输出 `{ kind: "data", subject: "help", data: { commands: UiCommandSpec[] } }`——扁平的 `commands` 数组，无分类。

### TUI 格式化 (events.ts:668-759)
`formatDataCommandOutput()` 使用 helper 函数（`getString`, `getRecord`, `formatModelLabel`）处理数据：
```ts
function formatDataCommandOutput(
  output: Extract<UiCommandOutput, { readonly kind: "data" }>,
): string {
  switch (output.subject) {
    case "status": {
      const status = getString(output.data, "status");
      const model = formatModelLabel(getRecord(output.data, "model"));
      if (status && model) return `status: ${status} | model: ${model}`;
      return status ? `status: ${status}` : JSON.stringify(output.data);
    }
    case "tools": { /* 提取 tool.name 数组 */ }
    case "session.list": { /* 提取 session title/id */ }
    // ... 其他 case ...
    default: return JSON.stringify(output.data);  // ← /help 走这里
  }
}
```
**结论**：`/help` 输出的是整个 `UiCommandCatalog` 的 JSON，在 TUI 中展示为一行 240 字符截断的乱码（`truncateCommandOutput` 在 events.ts:762 截断）。

## 1.8 关键约束

1. **事件总线输出模式**：所有命令通过 `context.emitOutput(dataOutput(subject, data))` 输出，TUI 通过 `formatDataCommandOutput()` 统一格式化 —— 这是现有架构的核心约束，不应破坏
2. **Provider DI 模式**：`CommandServiceOptions`（16 个字段）是依赖注入容器，所有外部能力通过 optional callback 注入 —— 新增能力应遵循此模式
3. **Surface 感知**：命令需声明在哪些 surface 可用（`"tui" | "stdout" | "headless" | "remote" | (string & {})`）
4. **Handler 与 Catalog 解耦**：命令 spec 在 catalog.ts，handler 在 builtin.ts（standalone function），通过 `createBuiltinHandlers()` 包装为 `CommandHandler` 对象，以 `commandId` 关联
5. **扩展点**：`extraCommands` + `extraHandlers` 允许外部注入命令，无需修改 catalog/builtin；`isAllowedExternalCommand()` 保护保留路径（`cancel`, `mode`, `model`, `permission/default`, `permission/full-access`）
6. **`permission.toggle-mode` 特殊模式**：handler 存在但无 catalog 条目，仅由 Shift+Tab 触发，是 handler/catalog 解耦的边界案例
7. **输出截断**：`truncateCommandOutput()` (events.ts:762) 将输出截断至 `COMMAND_NOTICE_TEXT_LIMIT`（240 字符），长列表输出会被截断
