## 问题 1：`/new` 之后终端不清屏

### 现象

执行 `/new` 后，OHBABY logo 和空输入框会正确出现在终端底部。但**之前 session 的全部消息仍可在上方的滚动回滚缓冲区看到**，视觉上没有"新开一个窗口"的感觉。

### 根因分析

`/new` 的完整调用链技术上是正确的：

```
/new
 └─ handleSessionNew()                     builtin.ts:442
     └─ createSessionFromCommand()          ui-inprocess.ts:651
         ├─ stateStore.setActiveSessionId(newId)
         ├─ publish("session.updated", ...)
         └─ publish("snapshot.replaced", ...)
              └─ TUI store: messages = [], activeSessionId 变化
                   └─ app.tsx:335
                        <TranscriptViewport key={activeSessionId ?? "none"}>
                             ↑ key 变化 → React 重挂整棵转录树 → transcript 干净
```

**但有一个根本限制**：Ink 的渲染机制（基于 `log-update`）只管理它自己的"动态区"，即当前帧的内容。**终端的 scrollback（历史行缓冲区）不归 Ink 管**。上一个 session 的输出已经被写入终端的历史行，Ink 无法撤回它们。

React 的 `key` 机制只保证新 session 的转录区是空的，并不等于清空了整个终端屏幕。

### 需要的修复

在切换到新建的空 session 时，主动向 `process.stdout` 写入 ANSI 清屏转义序列，清除终端屏幕和滚动缓冲并将光标移回左上角：

```
\x1b[2J    清除整个屏幕
\x1b[H     光标移到左上角（第 1 行第 1 列）
```

**触发时机**：在 `app.tsx` 中检测到 `activeSessionId` 切换到一个 `messages.length === 0` 的新 session 时执行。需要注意只在 `/new` 这种"真正新建"场景触发，而不是 `/resume` 切换到已有历史 session 时触发（否则反而会清掉用户本来想看的内容）。

**区分方式**：可通过 `command.result.delivered` 事件的 `action.kind === "session.selected"` + 新 session 消息数为零来联合判定，或后端 `/new` 命令直接在 action 里加 `kind: "session.new.created"` 标记。

### 涉及文件

- `packages/ohbaby-cli/src/tui/app.tsx`：在事件处理或 `useEffect` 中检测新空 session 并写清屏序列
- `packages/ohbaby-agent/src/commands/builtin.ts`：（可选）`handleSessionNew` 的 action 中加明确标记
- `packages/ohbaby-agent/src/commands/catalog.ts`：（可选）若需要区分 new vs resume

## 问题 2：`/new` 无限制创建空会话

### 现象

连续执行 `/new` 多次，或误触 `/new`，会产生多个标题完全相同（"New session"）、没有任何消息的空会话。这些空会话在 `/sessions` 列表里堆积，没有任何价值，污染 session 历史。

### 根因分析

四层调用链逐层无防护：

```
/new
 └─ handleSessionNew()
      if (!options.sessions?.createSession) { fail; return; }
      const session = await options.sessions.createSession();  // ← 无任何空会话检查
      ...

 └─ createSessionFromCommand()    ui-inprocess.ts:651
      await reserveIdsFromState();
      const session = { id: ..., messages: [], title: "New session" };
      await upsertSession(session);                            // ← 直接写入
      await stateStore.setActiveSessionId(session.id);
      ...

 └─ sessionManager.create()       manager.ts:100
      await options.store.insert(session);                    // ← 只检查 ID 重复
```

`store.insert()` 仅有 ID 唯一性校验（`DuplicateSessionError`），完全不检查消息数量。整条链路没有任何"是否已有空会话"的判断。

### 设计决策（已拍板）

**仅保持一个空会话**：若当前活跃 session 本身已经是空的（`messages.length === 0`），执行 `/new` 应该静默复用当前空 session，不再额外创建新 session，也不产生任何通知。

**不检查全局所有空 session**：只检查"当前活跃 session 是否为空"即可，避免跨 project 的空 session 被误命中；且用户切换 project 时合理的新空 session 应当被允许。

### 需要的修复

在 `createSessionFromCommand`（`packages/ohbaby-agent/src/adapters/ui-inprocess.ts:651`）函数体最前端加短路判断：

```typescript
async function createSessionFromCommand(): Promise<CommandSessionSummary> {
  // 若当前活跃 session 已是空的，静默复用，不创建新 session
  const snapshot = await stateStore.readSnapshot();
  const active = snapshot.sessions.find(s => s.id === snapshot.activeSessionId);
  if (active && active.messages.length === 0) {
    // 不发任何事件，不做任何副作用，直接返回
    return { id: active.id, title: active.title };
  }

  // 原有创建逻辑...
  await reserveIdsFromState();
  ...
}
```

**注意**：`handleSessionNew`（`builtin.ts:442`）仍然会 `emitOutput("session.created", ...)` 和 `emitAction("session.selected", ...)`，但这两个事件在复用路径下会指向同一个已有 session，不会产生副作用。TUI 的 `shouldDisplayCommandOutput` 已对 `"session.created"` 做了抑制，不会出现多余通知。

### 涉及文件

- `packages/ohbaby-agent/src/adapters/ui-inprocess.ts`：`createSessionFromCommand` 函数头部加短路逻辑（~L651）

## 问题 3：输入框宽度被 132 列硬上限截断

### 现象

在宽屏终端（Windows Terminal 全屏，约 155–185 列宽）下，输入框右侧有明显的大量空白区域未被利用。

### 根因分析

`metrics.ts` 中 `contentWidth` 计算有一个 `132` 的硬上限：

```typescript
// packages/ohbaby-cli/src/tui/layout/metrics.ts:17-20
const horizontalPadding = isCompact ? 2 : 4;  // 正常终端下为 4
const contentWidth = Math.min(
  132,    // ← 硬上限，导致宽屏资源浪费
  Math.max(24, columns - horizontalPadding * 2),
);
```

`Prompt` 组件的外层 Box 固定使用 `width={layout.contentWidth}`：

```tsx
// packages/ohbaby-cli/src/tui/components/prompt/index.tsx:213-219
<Box
  borderColor={theme.border}
  borderStyle="round"   // 左右各占 1 列
  flexDirection="column"
  paddingX={1}          // 左右各占 1 列
  width={layout.contentWidth}  // 锁死为 contentWidth，不自适应
>
```

**在 155 列终端下的实际列宽消耗：**

| 位置 | 列数 |
|---|---|
| AppShell 左侧内边距 | 4 |
| contentWidth（被 cap 截至 132） | 132 |
| AppShell 右侧**闲置** | 155 − 4 − 132 = **19** |
| 圆角边框（左右各 1） | 2 |
| 输入框内边距 paddingX（左右各 1） | 2 |
| **实际可打字宽度** | **128 列** |

180 列终端下浪费更严重：右侧闲置 40 列，实际可打字仍只有 128 列。

### 设计决策（已拍板）

提升 `contentWidth` 的上限，让输入框在宽屏下充分利用终端宽度。上限不应完全移除（避免超宽屏下内容行过长影响可读性），调整为：

- 将 `132` 上限改为 `160`（在 160 列终端下刚好不截断，180 列下只损失 12 列）
- 同时将 `prompt/index.tsx` 中 `width={layout.contentWidth}` 改为 `width="100%"`，使输入框宽度跟随父容器弹性变化，而不是硬编码为 `contentWidth` 的数字

**不调整 `horizontalPadding`**（保持 4），因为 4 列的左右留白对可读性是有益的，不是主要问题所在。

### 涉及文件

- `packages/ohbaby-cli/src/tui/layout/metrics.ts`：第 17 行 `132` 改为 `160`
- `packages/ohbaby-cli/src/tui/components/prompt/index.tsx`：Prompt Box 的 `width={layout.contentWidth}` 改为 `width="100%"`
- `packages/ohbaby-cli/src/tui/layout/metrics.unit.test.ts`：对应更新 132 上限相关的测试断言

## 问题 4：工具调用颜色与用户消息背景颜色设计

### 现象 A：工具调用文字色使用天蓝，与 logo 主色调不呼应

工具调用行（工具名 + 参数摘要）整行以天蓝色显示。天蓝在品牌语义里是"信息/链接"色，作为**主动操作**的工具调用用天蓝色在感知上略显"信息性"而非"行动性"。

此外还发现一个实现缺陷：`theme.tool.arg`（暗灰色，为参数摘要次要信息设计）在 token 层有定义，但**在整个组件代码里从未被引用**。`message-row.tsx` 的 `pairedPartColor` 函数对工具行只用了 `theme.tool.name`，导致工具名和参数摘要使用同一颜色，没有主次区分。

```typescript
// packages/ohbaby-cli/src/tui/components/message/message-row.tsx
function pairedPartColor(part, theme) {
  if (part.kind === "tool") {
    return part.call.status === "failed" || part.result?.error
      ? theme.tool.failed
      : theme.tool.name;  // ← 整行（名 + 参数）都用 tool.name，tool.arg 从未使用
  }
  ...
}
```

而 `renderToolLabel`（`tool-part.tsx:15`）把工具名和参数摘要拼接为一个字符串返回，导致无法在外部对两者单独着色。

### 现象 B：用户历史消息背景在暗色模式下与纯黑背景区分度极低

用户历史消息的背景色 `userBlockBg` 在暗色模式为 `#122238`（极深海军蓝，R18 G34 B56），与典型暗色终端背景（`#0C0C0C` 或 `#1E1E1E`）对比度极低，视觉上几乎无法感知背景的存在，失去了"用户消息区别于 AI 回复"的设计目的。

### 根因

**现象 A**：`tokens.ts` 中 `tool.name` 映射到 `skyBlue`，设计时将"工具名"定为天蓝，但工具行同时承载了行动语义；`tool.arg` token 从未落实到渲染层。

**现象 B**：`colors.ts` 中 `darkPalette.userBlockBg = "#122238"` 选色偏暗，在纯黑终端背景下区分度不足。这是暗色调色板的选色问题。

### 设计决策（已拍板）

#### 工具调用颜色

**工具名改用 logo 金色**，以品牌主色（金铠甲）代表"主动行动/工具调用"，天蓝继续用于"信息/链接"语义，二者不再重叠：

| token | 当前 | 调整后 | 语义 |
|---|---|---|---|
| `tool.name` | `skyBlue` | **`gold`** | 工具名（主动行动，品牌主色） |
| `tool.arg` | `textDim`（已定义，从未用） | **`textDim`（实际落实）** | 参数摘要（次要，暗灰） |

同时需修复 `tool.arg` 从未被使用的问题：将 `renderToolLabel` 的单字符串输出拆为两段分别着色，或返回结构化对象，使工具名和参数摘要能独立设置颜色。

**调色板变更（仅 `tokens.ts` 映射变更，`colors.ts` 无需改动）：**

```typescript
// tokens.ts — 修改前
tool: {
  name: color("skyBlue", "cyan"),   // 天蓝
  arg:  color("textDim", "gray"),   // 暗灰（已定义但从未使用）
  ...
}

// tokens.ts — 修改后
tool: {
  name: color("gold", "yellow"),    // 金色（logo 主色）
  arg:  color("textDim", "gray"),   // 暗灰（正式落实到渲染层）
  ...
}
```

#### 用户消息背景

**暗色模式 `userBlockBg` 改用更亮的天蓝**，让用户消息背景与 AI 回复的黑色背景形成清晰对比：

| 模式 | 当前值 | 调整后 | 变化 |
|---|---|---|---|
| 暗色 | `#122238`（极深海军蓝，几乎与黑背景混同） | **`#1E4472`**（深天蓝，明显可辨的蓝色底） | 亮度提升约 3 档，天蓝家族清晰可辨 |
| 亮色 | `#DCEEFF`（浅天蓝，对比度足够） | **保持不变** | 亮色主题无此问题 |

**`#1E4472` 选色依据**：
- R30 G68 B114：明确属于天蓝色系（与 logo 天蓝 `#6E9FCE` 同色族，但亮度降至适合背景使用的层次）
- 在纯黑终端（`#0C0C0C`）背景下，肉眼可清晰辨认为"有蓝色底"
- 暖白正文（`#E8E4DC`）在此背景上的对比度约为 7:1，可读性良好
- 不与品牌紫（`#B9A3E3`）混淆，与天蓝家族保持一致

**`colors.ts` 变更：**

```typescript
// 暗色调色板
export const darkPalette = {
  ...
  userBlockBg: "#1E4472",   // 原 "#122238" → 更亮的深天蓝，暗色下有明显区分度
  ...
} as const satisfies RawPalette;

// 亮色调色板：不变
export const lightPalette = {
  ...
  userBlockBg: "#DCEEFF",   // 保持不变
  ...
} as const satisfies RawPalette;
```

### 涉及文件

**工具颜色修复（两处）：**
- `packages/ohbaby-cli/src/tui/theme/tokens.ts`：`tool.name` 的 palette key 从 `"skyBlue"` 改为 `"gold"`
- `packages/ohbaby-cli/src/tui/components/message/message-row.tsx` + `parts/tool-part.tsx`：拆分 `renderToolLabel` 输出，使工具名（金色）和参数摘要（暗灰）可分别着色

**用户消息背景修复（一处）：**
- `packages/ohbaby-cli/src/tui/theme/colors.ts`：`darkPalette.userBlockBg` 从 `"#122238"` 改为 `"#1E4472"`

**需同步更新的测试：**
- `packages/ohbaby-cli/src/tui/theme/tokens.unit.test.ts`：`tool.name` 颜色断言从 `skyBlue` 改为 `gold`
- `packages/ohbaby-cli/src/tui/theme/colors.unit.test.ts`：`darkPalette.userBlockBg` 断言新值
- `packages/ohbaby-cli/src/tui/theme/tokens.unit.test.ts`：`message.userBlockBg` 相关断言（当前测试断言它不等于 `surface` 和 `border`，这两点仍成立，无需改）
- `packages/ohbaby-cli/src/tui/components/message/message-row.unit.test.tsx`：工具行颜色相关断言
