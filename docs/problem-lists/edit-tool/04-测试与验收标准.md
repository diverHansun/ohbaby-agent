# 4. 测试与验收标准

## 4.0 SWE 原则声明

> **测试的目的是验证行为正确性，而非制造"通过的假象"。**
>
> - 不能为了通过测试而测试——每个测试必须验证真实的用户场景
> - 不能通过放宽断言来"修复"失败——必须找出根因并修正代码
> - 不能只测试 happy path——边界情况、并发场景、错误恢复都要覆盖
> - 验收标准必须可测量、可重现、可回归
>
> 参考：IEEE 829 测试文档标准，ISTQB 测试原则（测试显示缺陷的存在而非缺席）

| ID | 场景 | 预期行为 | 验证方式 |
|----|------|----------|----------|

## 4.2 方案B 验收标准

### 4.2.1 功能验收

除方案A 的 A1-A4 外，增加：

| ID | 场景 | 预期行为 | 验证方式 |
|----|------|----------|----------|
| B1 | read → edit → edit（无 `expected_mtime_ms`） | 第二次 edit 因缺少 `expected_mtime_ms` 被拒绝 | 单元测试 |
| B2 | read → edit（正确 expected_mtime_ms）→ edit（正确 expected_mtime_ms） | 两次都成功 | 集成测试 |
| B3 | read → edit（错误 expected_mtime_ms） | 被 `assertExpectedMtimeMs` 拒绝 | 单元测试 |
| B4 | edit 未先 read（无 expected_mtime_ms） | 报 "expected_mtime_ms is required" | 单元测试 |
| B5 | read → 外部程序修改文件 → edit（旧 expected_mtime_ms） | mtime 不匹配，被拒绝 | 集成测试 |
| B6 | read → edit（格式化保存修改 mtime）→ edit（新 expected_mtime_ms） | 格式化后 mtime 变了但内容未变，第二次 edit 通过（因为 expected_mtime_ms 来源于第一次 edit 的返回值） | 集成测试 |

### 4.2.2 删除的测试用例

以下测试用例可以移除（因为被删除的代码不再存在）：

- `"requires the file to be read in the same session before edit"`（`edit.unit.test.ts:154-171`）
- 过期 mtime 拒绝中涉及 `assertTextFileWasReadBeforeEdit` 的部分（`edit.unit.test.ts:200-269`）

**删除理由**：被测代码已移除，测试不可编译/执行。这是合理的测试清理，不是"为了通过测试而删测试"。

## 4.3 方案C 验收标准

### 4.3.1 功能验收

| ID | 场景 | 预期行为 |
|----|------|----------|
| C1 | read → edit → edit（连续编辑） | 两次都成功（无需 mtimeMs） |
| C2 | edit 未先 read（但 old_string 碰巧正确） | 成功（放宽约束） |
| C3 | read → 外部程序修改文件 → edit | old_string 不匹配，报 "old_string not found" |
| C4 | read → edit（格式整）→ edit | 格式整导致空白变化，模糊匹配仍成功 |
| C5 | 两个并发 edit 到同一文件不同位置 | 都成功，且两次修改都生效 |
| C6 | 两个并发 edit 到同一文件同一位置 | 第一个成功，第二个 old_string 找不到 |
