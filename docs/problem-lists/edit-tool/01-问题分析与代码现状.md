# 1. 当前问题分析与代码现状

## 1.1 问题现象

Agent 连续对同一文件执行两次 `edit` 操作时，第二次报错：

```
File must be read again before edit: <path>.
The file changed since the last read.
```

Agent 被迫在每次 edit 后重新 read，导致工具调用链不连贯、token 消耗增加、交互效率降低。

### 1.2.2 双重检查机制

`edit.ts`（`packages/ohbaby-agent/src/tools/edit.ts:62-72`）在写入前执行两次独立检查：

| 检查 | 函数 | 位置 | 数据流 |
|------|------|------|--------|
| #1 | `assertExpectedMtimeMs` | `text-files.ts:233-248` | LLM传入的 `expected_mtime_ms` vs `fs.stat()` 实际值 |
| #2 | `assertTextFileWasReadBeforeEdit` | `read-state.ts:33-47` | 全局 `readFileStates` Map vs `fs.stat()` 实际值 |

### 1.2.3 mtimeMs 的产生与传递链路

```
read.ts:49
  └─ readTextFileContent(resolvedPath)
       └─ fs.stat() → mtimeMs (OS 文件修改时间)
            ├─ 存入全局 Map: recordTextFileRead()   [read.ts:50-54]
            └─ 返回给 LLM: metadata.mtimeMs          [read.ts:67]

edit.ts:65
  └─ readTextFileContent(existingPath)
       └─ fs.stat() → 当前文件 mtimeMs
            ├─ CHECK #1: assertExpectedMtimeMs(LLM传入值, 当前值)   [edit.ts:66]
            └─ CHECK #2: assertTextFileWasReadBeforeEdit(当前值)     [edit.ts:67-72]
                            └─ 查全局 Map: readFileStates.get(sessionId\x00path)
                               └─ 比较 state.mtimeMs !== actualMtimeMs → 报错

edit.ts:119-120
  └─ writeTextFileAtomic() → 文件 mtimeMs 变为新值
       └─ readWrittenFileMetadata() → 读取新 mtimeMs
            └─ 返回给 LLM: metadata.mtimeMs          [edit.ts:130]
            └─ ⚠ 未调用 recordTextFileRead()！        ← BUG
```

### 1.2.4 全局读取状态追踪器

`read-state.ts` 的实现（`packages/ohbaby-agent/src/tools/utils/read-state.ts`）：

```typescript
const MAX_READ_FILE_STATES = 10_000;
const readFileStates = new Map<string, ReadFileState>(); // 模块级单例

function keyFor(context: ToolExecutionContext, resolvedPath: string): string {
  return `${context.sessionId}\0${resolvedPath}`;
}

export function recordTextFileRead(input: {
  readonly context: ToolExecutionContext;
  readonly mtimeMs: number;
  readonly resolvedPath: string;
}): void {
  // FIFO 淘汰，超过 10,000 条时移除最早条目
  readFileStates.set(keyFor(input.context, input.resolvedPath), {
    mtimeMs: input.mtimeMs,
  });
}

export function assertTextFileWasReadBeforeEdit(input: {
  readonly actualMtimeMs: number;
  readonly context: ToolExecutionContext;
  readonly resolvedPath: string;
}): void {
  const state = readFileStates.get(keyFor(input.context, input.resolvedPath));
  if (!state) {
    throw new Error("File must be read before edit: ...");
  }
  if (state.mtimeMs !== input.actualMtimeMs) {
    throw new Error("File must be read again before edit: ...");
  }
}
```

关键特征：
- **模块级 `Map`**，进程生命周期内全局共享
- **仅 `read.ts` 写入**此 Map（`read.ts:50-54`）
- **仅 `edit.ts` 读取**此 Map（通过 `assertTextFileWasReadBeforeEdit`）
- **无 TTL、无失效机制**，一旦存入就永驻，直到 10,000 条 FIFO 淘汰
- **`write.ts` 完全不参与**此 Map 的读写

## 1.3 Bug 根因分析

### 完整链路：read → edit → edit

**Step 1 — Read：**
- `fs.stat()` → `mtimeMs = T0`
- `recordTextFileRead({mtimeMs: T0})` → Map 存入 `{session\x00path → T0}`
- 返回 LLM: `metadata.mtimeMs = T0`

**Step 2 — First Edit：**
- LLM 传入 `expected_mtime_ms = T0`
- `fs.stat()` → `mtimeMs = T0`（文件未变）
- CHECK #1: `T0 === T0` ✅
- CHECK #2: Map 中 `T0 === T0` ✅
- `writeTextFileAtomic()` → 文件时间戳变为 `T1`
- `readWrittenFileMetadata()` → 读取新值 `T1`
- 返回 LLM: `metadata.mtimeMs = T1`
- ⚠ **`recordTextFileRead()` 未被调用**，Map 仍为 `{session\x00path → T0}`

**Step 3 — Second Edit（失败点）：**
- LLM 传入 `expected_mtime_ms = T1`（来自 Step 2 返回值）
- `fs.stat()` → `mtimeMs = T1`
- CHECK #1: `T1 === T1` ✅（能通过）
- CHECK #2: Map 中 `T0 !== T1` → ❌ **`"File must be read again before edit"`**

### 根因总结

两个层面的问题：

1. **直接原因**：`edit.ts:119-130` 写入后读取了新 `mtimeMs` 并返回给 LLM，但**从未调用 `recordTextFileRead()`** 更新全局 Map。同样的问题存在于 `write.ts`。

2. **设计原因**：检查 #2（`assertTextFileWasReadBeforeEdit`）与检查 #1（`assertExpectedMtimeMs`）功能**高度重叠**，但使用了不同步的数据源，形成了一种虚假的安全冗余——实际上一方失效就导致整个流程卡死。
