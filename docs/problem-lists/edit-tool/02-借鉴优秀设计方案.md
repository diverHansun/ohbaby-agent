# 2. 借鉴优秀设计方案

## 2.1 对比总览

| 维度 | ohbaby-agent（当前） | kimi-code | opencode |
|------|---------------------|-----------|----------|
| 安全机制 | mtimeMs 时间戳双重检查 | old_string 内容匹配 | old_string 内容匹配（10级模糊） |
| 状态管理 | 全局 Map（session+path → mtimeMs） | 无状态 | 无状态 |
| 并发控制 | 无 | ToolAccesses 读写冲突序列化 | 文件级 Semaphore 锁 |
| 连续编辑 | ❌ 有 bug | ✅ 每次重新读取文件 | ✅ 锁后重新读取 |
| 外部修改检测 | mtimeMs 变化 | old_string 匹配失败 | old_string 匹配失败 |
| 虚假阳性 | 是（格式化保存等导致 mtime 变化） | 否（内容不匹配才报错） | 否（10级模糊匹配进一步降低） |

## 2.2 kimi-code 的设计（`D:\Projects\Code-cli\kimi-code`）

### 2.2.1 核心理念：内容即真理

kimi-code 的设计哲学是**完全无状态的、基于内容的验证**。系统不追踪任何文件读取状态，不做任何时间戳比较。每次 edit 调用自给自足。

### 2.2.2 Edit 工具实现要点

文件：`packages/agent-core/src/tools/builtin/file/edit.ts`（149行）

```
1. 读取参数: { path, old_string, new_string, replace_all? }
2. 自己读取当前文件内容: kaos.readText(safePath)        ← 每次 edit 内部重新读取
3. 内容匹配:
   - replace_all=false: 查找 old_string 出现次数
     · 0次 → "old_string not found, may be out of date"
     · 多次 → "old_string is not unique, include more context"
     · 1次 → 替换
   - replace_all=true: split + join，找不到则报错
4. 写入: kaos.writeText()
```

关键设计决策：
- **edit 永远读取文件的最新内容**，不依赖任何先前读取的缓存
- **唯一的安全检查是 old_string 是否出现在当前文件中**
- 如果旧内容已过时（被其他操作修改），`old_string` 自然找不到
- 这天然支持连续编辑，因为每次 edit 都从文件系统读最新状态

### 2.2.3 资源访问序列化

文件：`packages/agent-core/src/loop/tool-access.ts`（107-123行）

```typescript
function fileOperationsConflict(left, right): boolean {
  return fileOperationWrites(left) || fileOperationWrites(right);
}
// 'write' 和 'readwrite' 都视为"写入"
// 所以同一文件的编辑会被 ToolScheduler 自动序列化
```

### 2.2.4 可借鉴的设计要素

| 要素 | 说明 |
|------|------|
| **无状态编辑** | edit 工具自己读取最新文件内容，不依赖外部状态 |
| **内容匹配即验证** | old_string 找不到 = 内容已过时，自然拒绝编辑 |
| **资源访问声明** | ToolAccesses 声明 read/write/readwrite，调度器据此序列化冲突操作 |
| **提示工程兜底** | edit.md 描述告诉模型 "base every old_string on the latest Read" |

## 2.3 opencode 的设计（`D:\Projects\Code-cli\opencode`）

### 2.3.1 核心理念：锁 + 模糊匹配

opencode 在 kimi-code 的基础上增加两个关键设计：
1. **文件级并发锁**（Semaphore）确保编辑不交错
2. **10 级模糊匹配链**大幅降低"old_string not found"的误报率

### 2.3.2 文件级 Semaphore 锁

文件：`packages/opencode/src/tool/edit.ts:35-45`

```typescript
const locks = new Map<string, Semaphore.Semaphore>()

function lock(filePath: string) {
  const resolvedFilePath = AppFileSystem.resolve(filePath)
  const hit = locks.get(resolvedFilePath)
  if (hit) return hit
  const next = Semaphore.makeUnsafe(1)  // 容量=1 → 互斥
  locks.set(resolvedFilePath, next)
  return next
}

// 使用方式:
lock(filePath).withPermits(1)(() => executeEdit(...))
```

**为什么需要锁**：当 Agent 在一次响应中发出两个对同一文件的 edit 指令时，如果没有锁：
- Edit A 读取文件 → Edit B 读取文件（读到同样的内容）→ Edit A 写入 → Edit B 写入（覆盖 A 的修改）
- 有锁后：Edit A 执行完 → Edit B 拿到锁 → 重新读取（已包含 A 的修改）→ 执行

### 2.3.3 10 级模糊匹配链

文件：`packages/opencode/src/tool/edit.ts:672-710`

| 级别 | 匹配器 | 算法 |
|------|--------|------|
| 1 | SimpleReplacer | 精确字符串匹配 |
| 2 | LineTrimmedReplacer | 每行去除前后空格 |
| 3 | BlockAnchorReplacer | 首尾行锚点 + 中间行 Levenshtein 相似度 |
| 4 | WhitespaceNormalizedReplacer | 所有空白压成单个空格 |
| 5 | IndentationFlexibleReplacer | 去掉最小缩进 |
| 6 | EscapeNormalizedReplacer | 反转义 `\n` `\t` `\r` 等 |
| 7 | TrimmedBoundaryReplacer | 去掉首尾空白 |
| 8 | ContextAwareReplacer | 首尾行锚点 + 中间行 >=50% 相似度 |
| 9 | MultiOccurrenceReplacer | 精确全部匹配（用于 replaceAll） |
| 10 | 报错 | "Could not find oldString" |

前 4 级中若找到唯一匹配即停止。阈值：
- `SINGLE_CANDIDATE_SIMILARITY_THRESHOLD = 0.0`（唯一匹配零门槛）
- `MULTIPLE_CANDIDATES_SIMILARITY_THRESHOLD = 0.3`（多匹配需 30% 相似度区分）

### 2.3.4 可借鉴的设计要素

| 要素 | 说明 |
|------|------|
| **文件级互斥锁** | 防止同一文件的并发编辑互相覆盖 |
| **模糊匹配链** | 容忍空白/缩进/转义差异，降低误拒绝 |
| **提示工程** | prompt 声称需要 read 但实际不强制——信任 LLM 配合 old_string 匹配 |
| **零时间戳追踪** | 完全不用 mtimeMs，不维护读取状态 |

## 2.4 关键设计洞察

### 2.4.1 mtimeMs 方案的固有问题

1. **时钟敏感**：不同操作系统、文件系统的 mtime 精度不同（秒级/毫秒级/纳秒级）
2. **虚假阳性**：格式化保存、LSP 自动修复、git checkout 都会改变 mtime，但不影响 old_string 匹配
3. **状态管理负担**：需要维护 `readFileStates` Map，增加了 bug 表面积
4. **功能冗余**：检查 mtime ≠ 验证内容完整性。`old_string` 匹配才是真正的"内容未被意外修改"的验证

### 2.4.2 两个参考项目的共同选择

kimi-code 和 opencode **都完全放弃了时间戳追踪**。这不是巧合——它们分别来自不同的团队（Moonshot AI / Anthropic），但得出了相同的结论：

> `old_string` 匹配就是最好的编辑安全检查。如果文件内容变了，old_string 自然找不到。如果文件内容没变，即使 mtime 变了（如格式化），编辑也应该继续。

### 2.4.3 如何保证"必须读过才能编辑"

两个项目都不在代码层面强制，而是依赖：
1. **LLM 指令**：prompt 中告诉 LLM 先 read 再 edit
2. **内容惩罚**：如果 LLM 不读就编，`old_string` 很可能匹配不上（因为 LLM 不知道文件实际内容）
3. **错误提示**：匹配失败时提示 LLM 重新读取

这是一种**行为经济学设计**而非**技术强制设计**——用 LLM 自身的利益（避免错误重试）引导正确行为。
