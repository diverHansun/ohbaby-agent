# LLM 与 Agent 的功能边界与集成设计

## 一、LLM 与 Agent 的功能边界在哪里？

### 1.1 何时触发 LLM vs Agent？

在 gemini-cli 中的决策逻辑：

```
用户请求
  │
  ├─ "简单任务"（找个函数、修个bug）
  │  └─> 直接调用 LLM（normal flow）
  │
  └─ "复杂任务"（系统重构、深度分析、多文件影响）
     └─> 触发 codebase_investigator Agent
         （专门的代理来做深度分析）
```

**关键触发条件（来自 prompts.ts）：**

- 任务涉及 "complex refactoring"
- 需要 "codebase exploration"
- 需要 "system-wide analysis"
- 与多个文件、模块、架构相关

在 system prompt 中明确指导（prompts.ts 第 170 行）：

```
When the task involves complex refactoring, codebase exploration or 
system-wide analysis, your first and primary tool must be 
'codebase_investigator'.
```

**注意：** 对于简单的、目标明确的搜索（如查找特定函数名、文件路径、变量声明），应该直接使用 `grep` 或 `glob` 工具，而不是调用 Agent。 
### 1.2 Agent 比单纯的 User→LLM→Answer 多了哪些步骤？

**LLM 直接调用（简单路径）：**

1. 用户提问 → LLM 单次调用
2. LLM 返回结果（可能调用工具）
3. 返回答案

（全程：1 次 LLM 调用）

**Agent 调用（复杂路径）：**

1. 用户请求 → 主 LLM 识别需要 Agent
2. 主 LLM 返回 `functionCall("codebase_investigator", { objective: "..." })`
3. 工具调度器识别这是 `SubagentToolWrapper`，创建 `SubagentInvocation`
4. 进入 Agent 独立执行循环：
   a. `AgentExecutor.create()` 初始化 Agent（创建独立的 GeminiChat 实例、工具集、system prompt）
   b. `AgentExecutor.run()` 执行 Agent 循环（最多 15 轮，5 分钟超时）：
      - Agent LLM 调用 → 思考（通过 `onActivity` 回调流式输出）
      - Agent 调用工具（read_file、glob、grep、ls 等只读工具）
      - 收集结果 → 构建下一轮消息 → 继续循环
   c. Agent 完成（通过输出 schema 验证，返回结构化的 JSON 报告）
5. `SubagentInvocation.execute()` 返回 `ToolResult` 给工具调度器
6. 工具结果作为 `functionResponse` 返回给主 LLM
7. 主 LLM 继续处理或返回给用户

（全程：Agent 内部多次循环 + 主 LLM 的后续调用）
### 1.3 Agent 的特殊之处

Agent 具有以下特征：

**1. 独立的 LLM 循环**

- Agent 拥有自己的 `GeminiChat` 实例（在 `AgentExecutor.create()` 中创建）
- Agent 可以多轮与 LLM 交互（通过 `AgentExecutor.run()` 中的 while 循环）
- 主 LLM 不知道 Agent 内部在做什么，只看到最终结果

**2. 受限的工具集**

`codebase_investigator` 只能使用只读工具（codebase-investigator.ts 第 85 行）：
- `ls` - 列出目录
- `read_file` - 读取文件
- `glob` - 文件模式匹配
- `grep` - 文本搜索

不能使用的工具：
- `edit` - 编辑文件
- `write_file` - 写入文件
- `shell` - 执行命令

（这是设计，保证代理只做分析，不修改代码）

**3. 独立的 system prompt**

`codebase_investigator` 有自己的详细 system prompt（codebase-investigator.ts 第 93-152 行），指导代理如何：
- 做深度分析
- 理解代码的 "why" 而不仅仅是 "what"
- 系统性探索代码库
- 提供结构化的分析报告

**4. 结构化的输出**

- Agent 必须返回符合 `CodebaseInvestigationReportSchema` 的 JSON 对象
- 包含：SummaryOfFindings、ExplorationTrace、RelevantLocations
- 通过 `processOutput` 方法格式化为字符串返回给主 LLM

**5. 受限的执行时间**

- 超时限制：5 分钟（`max_time_minutes: 5`）
- 最大 turn 限制：15 轮（`max_turns: 15`）
- 通过 `checkTermination()` 方法检查终止条件
### 1.4 为什么需要独立 Agent？

**主 LLM 不适合做深度分析，因为：**

- **Token 限制：** 深度分析产生大量输出（超过 context window）
- **不能有效迭代：** 主 LLM 关注整体任务，不能聚焦于单一复杂任务的细节
- **能力限制：** 主 LLM 需要处理多种任务（代码生成、工具调用、用户交互等）

**Agent 的优势：**

- **专注性：** 专注于单一复杂任务（代码库分析）
- **迭代能力：** 可以多轮迭代深化理解（最多 15 轮）
- **独立上下文：** 有自己的 context window，不会污染主 LLM 的上下文
- **结构化输出：** 输出结构化的报告给主 LLM，便于后续处理
- **资源隔离：** Agent 的 token 消耗和主 LLM 分离，避免相互影响
## 二、Agent 作为工具的集成方式

### 2.1 SubagentToolWrapper 的作用

在 gemini-cli 中，Agent 不是某个特殊的执行路径，而是被包装成一个标准的 `DeclarativeTool`：

**注册流程（config.ts 第 1565-1587 行）：**

```typescript
// 在 createToolRegistry() 中注册 Agent 为工具
if (this.getCodebaseInvestigatorSettings().enabled) {
  const definition = this.agentRegistry.getDefinition('codebase_investigator');
  if (definition) {
    const wrapper = new SubagentToolWrapper(
      definition,
      this,
      messageBusEnabled ? this.getMessageBus() : undefined,
    );
    registry.registerTool(wrapper);
  }
}
```

**SubagentToolWrapper 的关键特性（subagent-tool-wrapper.ts）：**

- 继承自 `BaseDeclarativeTool`，实现标准的工具接口
- 动态生成 JSON schema（基于 Agent 的 `inputConfig`）
- 工具类型为 `Kind.Think`，表示这是一个需要思考的工具
- 支持流式输出（`canUpdateOutput: true`）

**主 LLM 的视角：**

主 LLM 看到 `codebase_investigator` 就像看到其他工具一样：
- 工具名称：`codebase_investigator`
- 工具描述：Agent 的 `description`
- 参数 schema：从 `inputConfig` 动态生成
- 调用方式：标准的 `functionCall`

### 2.2 调用流程详解

```
主 LLM 决定调用 Agent
       ↓
主 LLM 返回：functionCall("codebase_investigator", { objective: "..." })
       ↓
CoreToolScheduler.schedule() 接收工具调用请求
       ↓
识别工具类型：SubagentToolWrapper
       ↓
SubagentToolWrapper.build() 创建 SubagentInvocation
       ↓
CoreToolScheduler 执行工具：invocation.execute(signal, updateOutput)
       ↓
SubagentInvocation.execute()（invocation.ts 第 75-136 行）
  ├─> 创建 onActivity 回调（用于流式输出 Agent 的思考过程）
  ├─> AgentExecutor.create(definition, config, onActivity)
  │   └─> 创建独立的 GeminiChat 实例
  │   └─> 设置 Agent 专用的 system prompt
  │   └─> 配置 Agent 受限的工具集（只读工具）
  ├─> executor.run(inputs, signal)
  │   └─> while 循环执行多个 turn（最多 15 轮）
  │       ├─> 检查终止条件（超时、最大轮数、abort）
  │       ├─> executeTurn() - Agent LLM 调用
  │       ├─> 处理工具调用（通过 Agent 自己的工具调度器）
  │       └─> 构建下一轮消息，继续循环
  └─> 返回 ToolResult（包含格式化的报告）
       ↓
CoreToolScheduler 将 ToolResult 转换为 functionResponse
       ↓
结果作为 functionResponse 返回给主 LLM
       ↓
主 LLM 继续处理（可能继续调用其他工具或返回最终答案）
```

**关键点：**

1. **工具调度器统一处理：** Agent 工具调用和其他工具调用使用相同的调度器（`CoreToolScheduler`）
2. **执行隔离：** Agent 的执行在 `SubagentInvocation.execute()` 中完成，对主 LLM 透明
3. **流式输出：** Agent 的思考过程通过 `updateOutput` 回调实时传递给 UI
4. **结果封装：** Agent 的输出被封装为 `ToolResult`，包含 `llmContent` 和 `returnDisplay`
## 三、完整的 Turn 流程

### 3.0 Turn 的定义

在 gemini-cli 中，**Turn 不是指单次用户消息，而是一次完整的交互周期**。

**Turn 的层次结构：**

1. **主 Turn（User Turn）：** 从用户输入到最终响应的完整周期
   - 可能包含多次 LLM 调用
   - 可能包含多个工具调用（包括 Agent）
   - 由 `GeminiClient.sendMessageStream()` 管理

2. **Agent Turn：** Agent 内部的单次 LLM 调用周期
   - Agent 在 `AgentExecutor.run()` 中执行多个 Agent Turn
   - 每个 Agent Turn 可能包含工具调用
   - 由 `AgentExecutor.executeTurn()` 管理

**关键区别：**
- 一个主 Turn 可能包含多个 LLM 调用（如果调用了工具）
- 一个主 Turn 可能包含一个 Agent 调用
- 一个 Agent 调用内部包含多个 Agent Turn（最多 15 个）

### 3.1 简单场景的 Turn 流程
```
┌─ User Input ─────────────────────────────────────┐
│  "修复 bug XYZ"                                   │
└──────────────────────────────────────────────────┘
            ↓
┌─ GeminiClient.sendMessageStream() ────────────────┐

│  事件流：Content, ToolCallRequest, Finished        │
└──────────────────────────────────────────────────┘
            ↓
┌─ CoreToolScheduler.schedule() ────────────────────┐
│  工具状态流转：                                    │
│  validating → awaiting_approval → executing        │
│  → success                                         │
```

### 3.3 Turn 的实际数据结构（基于 turn.ts）

在 gemini-cli 中，`Turn` 是一个类（不是接口），实际结构如下：

```typescript
export class Turn {
  // 待处理的工具调用请求
  readonly pendingToolCalls: ToolCallRequestInfo[] = [];
  
  // 所有 LLM 响应（用于调试和获取文本）
  private debugResponses: GenerateContentResponse[] = [];
  
  // 待处理的引用
  private pendingCitations = new Set<string>();
  
  // 完成原因
  finishReason: FinishReason | undefined = undefined;

  constructor(
    private readonly chat: GeminiChat,
    private readonly prompt_id: string,
  ) {}
}
```

**关键方法：**

- `run()`: 异步生成器，执行 Turn 并产生事件流
- `getDebugResponses()`: 获取所有 LLM 响应
- `getResponseText()`: 获取所有响应的文本内容

**注意：**

- Turn 不直接存储用户输入，而是通过 `chat` 对象访问历史
- Turn 不直接存储工具结果，工具结果通过事件流传递
- `pendingToolCalls` 只包含待处理的工具调用请求，已完成的工具调用通过事件流处理

**事件流类型（ServerGeminiStreamEvent）：**

- `Content`: LLM 生成的文本内容
- `ToolCallRequest`: 工具调用请求
- `ToolCallResponse`: 工具调用响应
- `Thought`: 思考过程（如果启用）
- `Finished`: Turn 完成
- `Error`: 错误
- `ChatCompressed`: 历史压缩
- `LoopDetected`: 检测到循环
- 等等
## 四、ohbaby-code 应该如何设计？

基于 gemini-cli 的实际实现，以下是设计建议：

### 4.1 Turn 的数据结构设计

**建议的 Turn 接口：**

```typescript
interface Turn {
  // 用户输入（可选，因为可以通过 history 获取）
  userMessage?: ChatCompletionMessage;

  // LLM 的所有调用和响应
  llmResponses: GenerateContentResponse[];

  // 所有工具调用请求（包括 Agent）
  toolCallRequests: ToolCallRequestInfo[];

  // 所有工具调用响应
  toolCallResponses: ToolCallResponseInfo[];

  // 事件流（用于流式处理和调试）
  events: ServerGeminiStreamEvent[];

  // 完成原因
  finishReason: FinishReason | undefined;

  // Metadata
  promptId: string;
  timestamp: Date;
}
```

**关键设计点：**

1. **不直接存储用户输入：** 用户输入可以通过 conversation history 获取
2. **分离工具请求和响应：** 便于追踪工具调用的生命周期
3. **包含事件流：** 支持流式处理和调试
4. **简化 Agent 处理：** Agent 调用在工具层面统一处理，不需要特殊字段

### 4.2 Client 的执行流程设计

**建议的 `sendMessageStream` 方法（基于 client.ts）：**

```typescript
async *sendMessageStream(
  request: PartListUnion,
  signal: AbortSignal,
  promptId: string,
  maxTurns: number = MAX_TURNS,
): AsyncGenerator<ServerGeminiStreamEvent, Turn> {
  // 1. 初始化 Turn
  const turn = new Turn(this.getChat(), promptId);

  // 2. 检查会话限制（最大 turn 数、循环检测等）
  if (this.sessionTurnCount > this.config.getMaxSessionTurns()) {
    yield { type: GeminiEventType.MaxSessionTurns };
    return turn;
  }

  // 3. 检查上下文窗口溢出
  const estimatedTokens = await calculateRequestTokenCount(...);
  if (estimatedTokens > remainingTokens * 0.95) {
    yield { type: GeminiEventType.ContextWindowWillOverflow, ... };
    return turn;
  }

  // 4. 尝试压缩历史（如果需要）
  const compressed = await this.tryCompressChat(promptId, false);
  if (compressed.compressionStatus === CompressionStatus.COMPRESSED) {
    yield { type: GeminiEventType.ChatCompressed, value: compressed };
  }

  // 5. 添加 IDE 上下文（如果启用）
  if (this.config.getIdeMode()) {
    const { contextParts } = this.getIdeContextParts(...);
    if (contextParts.length > 0) {
      this.getChat().addHistory({
        role: 'user',
        parts: [{ text: contextParts.join('\n') }],
      });
    }
  }

  // 6. 执行 Turn（产生事件流）
  const resultStream = turn.run({ model: modelToUse }, request, signal);
  for await (const event of resultStream) {
    // 检查循环
    if (this.loopDetector.addAndCheck(event)) {
      yield { type: GeminiEventType.LoopDetected };
      return turn;
    }
    yield event;
  }

  // 7. 如果没有待处理的工具调用，检查是否需要继续
  if (!turn.pendingToolCalls.length && !signal.aborted) {
    const nextSpeakerCheck = await checkNextSpeaker(...);
    if (nextSpeakerCheck?.next_speaker === 'model') {
      // 递归调用继续对话
      return yield* this.sendMessageStream(
        [{ text: 'Please continue.' }],
        signal,
        promptId,
        maxTurns - 1,
      );
    }
  }

  return turn;
}
```

**关键设计点：**

1. **异步生成器：** 使用 `async *` 支持流式事件输出
2. **递归处理：** 支持多轮对话（通过递归调用 `sendMessageStream`）
3. **工具调用处理：** 工具调用通过 `CoreToolScheduler` 统一处理，Agent 作为特殊工具
4. **事件驱动：** 所有状态变化通过事件流传递

### 4.3 工具调度器的设计

**工具调度的关键流程（基于 coreToolScheduler.ts）：**

```typescript
class CoreToolScheduler {
  async schedule(
    request: ToolCallRequestInfo | ToolCallRequestInfo[],
    signal: AbortSignal,
  ): Promise<void> {
    // 1. 验证工具是否存在
    const toolInstance = this.config.getToolRegistry().getTool(reqInfo.name);
    
    // 2. 构建工具调用（ToolInvocation）
    const invocation = toolInstance.build(reqInfo.args);
    
    // 3. 检查是否需要确认
    const confirmationDetails = await invocation.shouldConfirmExecute(signal);
    
    if (confirmationDetails) {
      // 4. 等待用户确认
      this.setStatusInternal(callId, 'awaiting_approval', signal, confirmationDetails);
    } else {
      // 5. 直接执行
      this.setStatusInternal(callId, 'scheduled', signal);
      await this.attemptExecutionOfScheduledCalls(signal);
    }
  }

  private async attemptExecutionOfScheduledCalls(signal: AbortSignal) {
    // 执行工具
    const toolResult = await invocation.execute(signal, updateOutput);
    
    // 如果是 SubagentInvocation，这里会触发 AgentExecutor
    // Agent 的执行对调度器透明
    
    // 转换结果为 functionResponse
    const response = convertToFunctionResponse(toolName, callId, toolResult.llmContent);
    
    // 更新状态
    this.setStatusInternal(callId, 'success', signal, response);
  }
}
```

**关键设计点：**

1. **统一接口：** Agent 工具和其他工具使用相同的调度流程
2. **状态管理：** 工具调用有明确的状态流转（validating → scheduled → executing → success/error）
3. **确认机制：** 支持工具调用前的用户确认
4. **透明执行：** Agent 的执行细节对调度器透明，只需要调用 `execute()` 方法

### 4.4 Agent 执行器的设计

**Agent 执行的关键流程（基于 executor.ts）：**

```typescript
class AgentExecutor {
  async run(inputs: AgentInputs, signal: AbortSignal): Promise<OutputObject> {
    // 1. 创建独立的 GeminiChat 实例
    const chat = await this.createChatObject(inputs, tools);
    
    // 2. 构建初始查询
    const query = templateString(this.definition.promptConfig.query, inputs);
    let currentMessage: Content = { role: 'user', parts: [{ text: query }] };
    
    // 3. Agent 循环（最多 max_turns 轮）
    while (true) {
      // 检查终止条件
      const reason = this.checkTermination(startTime, turnCounter);
      if (reason) break;
      
      // 执行单轮
      const turnResult = await this.executeTurn(
        chat,
        currentMessage,
        turnCounter++,
        signal,
      );
      
      if (turnResult.status === 'stop') {
        // 验证输出 schema
        const validatedOutput = this.definition.outputConfig.schema.parse(
          turnResult.finalResult
        );
        return {
          result: this.definition.processOutput(validatedOutput),
          terminate_reason: turnResult.terminateReason,
        };
      }
      
      // 继续下一轮
      currentMessage = turnResult.nextMessage;
    }
  }
}
```

**关键设计点：**

1. **独立上下文：** Agent 拥有自己的 `GeminiChat` 实例
2. **受限工具：** Agent 只能使用配置的工具集（通常只有只读工具）
3. **循环执行：** 通过 while 循环实现多轮对话
4. **结构化输出：** 输出必须符合预定义的 schema
5. **终止条件：** 支持超时、最大轮数、用户取消等终止条件

## 五、关键概念总结

### 5.1 核心设计原则

1. **Agent 即工具：** Agent 被包装成标准工具，主 LLM 可以像调用其他工具一样调用它
2. **执行隔离：** Agent 拥有独立的 LLM 实例、上下文和工具集，执行过程对主 LLM 透明
3. **统一调度：** 所有工具（包括 Agent）通过 `CoreToolScheduler` 统一调度和管理
4. **事件驱动：** 使用事件流（`ServerGeminiStreamEvent`）传递状态变化，支持流式处理

### 5.2 层次结构

```
User Turn (主 Turn)
  ├─> GeminiClient.sendMessageStream()
  ├─> Turn.run() - 主 LLM 调用 #1
  ├─> CoreToolScheduler.schedule()
  │   └─> SubagentInvocation.execute()
  │       └─> AgentExecutor.run()
  │           ├─> Agent Turn #1
  │           ├─> Agent Turn #2
  │           └─> Agent Turn #N (最多 15 轮)
  ├─> Turn.run() - 主 LLM 调用 #2
  └─> Turn Complete
```

### 5.3 关键组件

| 组件 | 职责 | 位置 |
|------|------|------|
| `GeminiClient` | 管理主对话流程，处理多轮对话 | `client.ts` |
| `Turn` | 表示一次完整的交互周期，管理事件流 | `turn.ts` |
| `CoreToolScheduler` | 统一调度所有工具调用（包括 Agent） | `coreToolScheduler.ts` |
| `SubagentToolWrapper` | 将 Agent 包装成标准工具 | `subagent-tool-wrapper.ts` |
| `SubagentInvocation` | Agent 工具的执行实例 | `invocation.ts` |
| `AgentExecutor` | 执行 Agent 的多轮循环 | `executor.ts` |
| `CodebaseInvestigatorAgent` | Agent 定义（工具集、prompt、配置） | `codebase-investigator.ts` |

### 5.4 数据流

```
用户输入
  ↓
GeminiClient.sendMessageStream()
  ↓
Turn.run() → 产生事件流
  ↓
LLM 响应（可能包含 functionCall）
  ↓
CoreToolScheduler.schedule()
  ↓
如果是 Agent 工具：
  SubagentInvocation.execute()
    ↓
  AgentExecutor.run()
    ↓
  多个 Agent Turn（LLM 调用 + 工具调用）
    ↓
  返回 ToolResult
  ↓
functionResponse 返回给主 LLM
  ↓
主 LLM 继续处理
  ↓
Turn Complete
```

### 5.5 设计优势

1. **模块化：** Agent 作为独立模块，易于扩展和维护
2. **可复用：** Agent 工具可以像其他工具一样被主 LLM 调用
3. **隔离性：** Agent 的执行不影响主 LLM 的上下文
4. **可观测：** 通过事件流可以追踪整个执行过程
5. **灵活性：** 支持不同的 Agent 配置（工具集、prompt、超时等）

### 5.6 实现建议

对于 ohbaby-code 的实现，建议遵循以下原则：

1. **保持一致性：** Agent 工具和其他工具使用相同的接口和调度机制
2. **事件驱动：** 使用事件流而非直接返回值，支持流式处理
3. **状态管理：** 明确的状态流转（validating → scheduled → executing → success/error）
4. **错误处理：** 统一的错误处理和恢复机制
5. **可扩展性：** 设计支持未来添加更多类型的 Agent
